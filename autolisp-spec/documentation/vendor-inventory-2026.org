* 1 Purpose

This document is the Phase-1 deliverable of the ongoing
work to bring [[file:autolisp-visual-lisp-specification-draft.org][AutoLISP Spec]]
to exhaustive coverage of the AutoLISP / Visual LISP language as
shipped by Autodesk AutoCAD 2026 and Bricsys BricsCAD V26.

It records:

- the canonical vendor symbol inventories that were extracted from
  AutoCAD 2026 and BricsCAD V26 vendor documentation,
- the per-symbol availability tags (AutoCAD-only / BricsCAD-only /
  both) at the level the vendor docs make explicit,
- the delta between those vendor inventories and the existing
  specification draft,
- a categorised list of missing entries that subsequent phases will
  fill in.

This document is *not* a normative part of the specification.

It is a working artefact tracking what the spec covers and what it
still needs to cover.

* 2 Scope and Method

** 2.1 Scope

In scope for the inventory:

- core AutoLISP and Visual LISP functions, special forms, and
  reader-syntax classes,
- DCL (Dialog Control Language) functions used from AutoLISP,
- ActiveX / COM bindings (~vlax-*~), reactors (~vlr-*~), and the
  Visual LISP family (~vl-*~),
- Express Tools functions (~acet-*~) and ~acad-*~ / ~acad_*~ helpers
  documented in the AutoLISP Reference,
- BricsCAD-specific extensions enumerated on the BricsCAD-Specific
  LISP Functions page,
- per-version flags where the vendor explicitly provides them.

Out of scope for this inventory (per project decision dated
2026-04-26):

- BricsCAD BIM, Mechanical, GeoCoordinateSystem, and SheetSetManager
  LISP APIs.
- Vertical / discipline-specific extensions are listed only as
  reference URLs.

** 2.2 Method

1. Fetched the AutoCAD 2026 "AutoLISP Functions By Name and Feature"
   index page.
2. Fetched all 24 alphabetical sub-indexes (Non-alphabetic, A-Z
   except J/K/Y) and enumerated every named entry.
3. Fetched the AutoCAD 2026 feature-category index links to confirm
   coverage of Express Tools, ActiveX, DCL, Reactors, and
   Selection/Symbol-Table groups.
4. Fetched the BricsCAD V26 BricsCAD-Specific LISP Functions page,
   the V26 Generic Properties Functions page, the V26 VLX Namespace
   Support page, the V26 read+write text files page, the V26
   ~vlisp-compile~ page, the V26 Optimised Code Patterns page, and
   the V26 LISP Optimiser pages.
5. Cross-checked against the V25 "What's New in BricsCAD Lisp" page
   and the V14/V9 LISP Compatibility pages where current pages
   returned 404.
6. Normalised every symbol name to lowercase for diffing.
7. Computed the diff against the 328 dictionary entries already
   present in the specification draft.

** 2.3 Caveats

- The Bricsys developer-reference index page is JavaScript-driven, so
  raw HTTP fetching of the framed TOC returns only the top-level
  category names. The full TOC tree is built at runtime from a
  ~chmtop.c2wtopf.TITEMS~ JavaScript array. Phase 1 was therefore
  re-run inside a real Chromium instance (via the Claude-in-Chrome
  MCP) on 2026-04-26: the live TITEMS tree was walked recursively
  and dumped to ~bricscad-lisp-toc.txt~ (1934 entries, 95.6 KB).
  This is the authoritative BricsCAD V26 LISP catalogue.
- The historical compatibility-table pages
  (~IDR_LISP_B100.htm~, ~CompatibilityWithAutoCAD.htm~,
  ~LISPCompatibility.htm~) returned 404 in V25 and V26; only the V9
  page is currently reachable. Per-symbol BricsCAD support for
  AutoCAD-documented AutoLISP names is now established by direct
  presence in the BricsCAD V26 TOC, not inferred — see §6 and §9.
- AutoCAD's localised reference URLs (FRA, KOR, CHT, ENU, ...) all
  resolve to the same content keyed by GUID. This inventory records
  ENU URLs only.

* 5 Per-Symbol Availability Model

For Phase 2 onward, every dictionary entry in the spec must carry an
explicit availability cell. The agreed-upon vocabulary is:

- ~AutoCAD~: documented on Autodesk's per-symbol page; not documented
  by Bricsys.
- ~BricsCAD~: documented by Bricsys; not documented by Autodesk.
- ~Both~: documented (or strongly implied as compatible) by both
  vendors.
- ~Both (presumed)~: in the AutoCAD AutoLISP reference and not
  contradicted by Bricsys, with no direct Bricsys page; default for
  AutoLISP core symbols pending a per-symbol confirmation.
- ~AutoCAD-Windows~: documented as Windows-only by Autodesk
  (ActiveX, ~acet-*~ Express Tools, several DCL helpers); BricsCAD
  status TBD.
- ~Both with divergence~: documented by both vendors, but with
  different signature, return shape, or platform footprint.

In the dictionary entries this becomes:

#+begin_example
*** Availability
- AutoCAD 2026: yes (Windows + macOS, except where noted).
- BricsCAD V26: yes (Windows + macOS + Linux).
- Status: documented in both.
#+end_example

* 10 Cross-Vendor Divergence Items Identified in Phase 1

The following points were noticed during inventory and must be
reflected in dictionary entries during Phase 3 / 4:

1. ~vlisp-compile~ has different mode arguments and different output
   formats:
   - Autodesk: produces ~.fas~, ~.vlx~ artefacts (per Autodesk
     reference page).
   - BricsCAD V26: produces ~.des~ encrypted output; the mode symbol
     ~'st / 'lsm / 'lsa~ is documented as accepted but ignored.
2. ~open~ accepts BricsCAD-extended ~ccs=~ encoding clauses
   (~"r,ccs=UNICODE"~, ~"r,ccs=UTF-8"~, ~"r,ccs=UTF-16LE"~,
   ~"w,ccs=UTF-8"~, ~"w,ccs=UTF-16LE"~) that are not part of the
   Autodesk ~open~ surface. Autodesk's Unicode story is governed by
   ~LISPSYS~ instead; BricsCAD's is per-call.
3. ~ssget~ and ~ssadd~ are documented as having BricsCAD-specific
   enhancements (V9-era compatibility page).
4. ~menucmd~ is documented in BricsCAD as supporting only the P0 and
   P1-P16 cursor / pulldown forms, not auxiliary, button, icon,
   diesel, screen, or tablet menus (V9-era compatibility page).
5. ~tan~ is BricsCAD-only. AutoCAD users compute it as
   ~(/ (sin x) (cos x))~.
6. AutoCAD's ~layerstate-*~ family and BricsCAD's ~vl-layerstates-*~
   family overlap in purpose but disagree on names; spec entries
   must cross-link.
7. AutoCAD's ~vl-position~ / ~vl-remove~ vs BricsCAD's ~position~ /
   ~remove~ overlap on functionality; cross-reference required.
8. The Autodesk ~vlax-typeinfo-available-p~ function is Windows-only;
   BricsCAD parity status is unknown (404 on the Bricsys
   compatibility table).
9. The Autodesk ~vl-vbaload~ / ~vl-vbarun~ pair is Windows-only and
   tied to AutoCAD's VBA module; BricsCAD does not document a
   parallel VBA surface.
10. ~showhtmlmodalwindow~ is Windows-only on Autodesk; BricsCAD has
    no documented equivalent.
11. *~or~ returns T / nil, not the first non-nil value.* Discovered
    during Phase-5 probe-suite preparation against BricsCAD V26
    (probe-core's ~(setq x (or x default))~ idiom silently clobbered
    ~x~ with ~T~). BricsCAD V26 documents this explicitly on the
    per-symbol page: "T if any provided argument evaluates as
    non-NIL; ... NIL otherwise". Autodesk's documentation is
    consistent with BricsCAD here. Ports from Common Lisp must
    rewrite ~or~-as-default-value idioms to use ~if~ instead. The
    spec entry for ~or~ now records the divergence and the portable
    idiom; ~probe-core.lsp~ uses the ~if~-guarded form.
12. *BricsCAD V26 ~atof~ accepts C99 hex-float syntax* such as
    ~"0x1p4"~ → ~16.0~. Tested 2026-04-26 against BricsCAD V26
    macOS via [[file:../scripts/run-probes-direct.sh][~scripts/run-probes-direct.sh~]]; result captured at
    [[file:../results/bricscad/macos/20260426T122808Z/results.sexp][~results/bricscad/macos/20260426T122808Z/results.sexp~]]. Locale
    sensitivity is *not* present on the macOS build (decimal-comma
    rejected). The conservative ~clautolisp~ lex model (no hex-float)
    is therefore strictly narrower than BricsCAD V26.
13. *~terpri~ is zero-arity in BricsCAD V26.* The 2-argument form
    ~(terpri stream)~ raises ~too few / too many arguments at
    [TERPRI]~. Use ~(princ "\n" stream)~ for portable file-handle
    newlines.
14. *The autolisp-script wrapper redefines printer functions.* The
    wrapper at ~/works/sncf-reseau/src/outils-autolisp/autolisp-script/
    redefines ~prin1~, ~princ~, ~print~, and ~prompt~ to single-
    argument helpers that mirror command-line output. Probes that
    need the 2-argument form (`prin1 expr stream`) must use the
    direct ~scripts/run-probes-direct.sh~ runner instead.
