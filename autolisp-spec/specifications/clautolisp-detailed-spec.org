* Executive Summary
This document describes the design of a portable AutoLISP runtime implemented in Common Lisp.

The goal is to allow legacy AutoLISP libraries to run offline outside CAD environments, while providing a modern development environment including debugging, testing, and regression validation.

The system is designed to run headless for automated testing and continuous integration pipelines. It also includes an optional UI layer implemented out of process using Qt6, enabling testable dialog logic without requiring a graphical environment.

* Strategic Objectives
- Provide a standalone interpreter capable of executing existing AutoLISP libraries.
- Enable CAD regression testing through deterministic drawing snapshots.
- Provide a modern AutoLISP development environment with debugging, inspection, and testing tools.
- Allow progressive migration from AutoLISP toward Common Lisp.
- Ensure portability across macOS, Linux, and Windows.

* System Architecture Overview
The system is composed of multiple subsystems organized around a layered architecture.

The interpreter core provides language semantics and runtime evaluation. Higher layers implement Visual LISP compatibility, UI abstraction, and host integration.

The modular architecture allows subsystems to evolve independently while maintaining stable interfaces.

* alisp-core Runtime
The ~alisp-core~ subsystem implements the AutoLISP language semantics.

It includes the reader, evaluator, symbol system, environment model, and primitive functions.

The runtime preserves AutoLISP behavior including dynamic scoping and symbol resolution rules.

** Core Components
- Reader: parses AutoLISP source expressions.
- Evaluator: executes forms using an explicit environment model.
- Environment model: stack of frames representing dynamic variable scope.
- Error system: mapped to Common Lisp conditions for debugging support.

* Visual LISP Compatibility Layer
The Visual LISP compatibility layer provides helper utilities typically available in modern AutoLISP implementations.

This includes string processing functions, file utilities, list manipulation helpers, and conversion utilities.

The functions are grouped into categories such as ~vl-string-*~, ~vl-file-*~, and ~vl-filename-*~.

* Dialog Control Language Subsystem
The DCL subsystem provides support for dialog-based user interfaces.

It implements parsing of DCL files and runtime creation of dialogs.

Tiles map to platform widgets through the UI abstraction layer.

* Out-of-Process UI Architecture
The UI layer is implemented as a separate process using Qt6.

This design enables the runtime to execute in headless environments while maintaining UI capabilities.

The runtime communicates with the UI process using a lightweight message protocol.

* CAD Host Integration
The host subsystem provides bridges to external CAD environments.

Host backends translate AutoLISP entity operations into calls to the target CAD API.

A mock backend allows regression tests without requiring a CAD installation.

* Snapshot-Based Regression Testing
Regression tests capture drawing state before and after AutoLISP execution.

Snapshots may be compared using multiple strategies including deep object comparison or entity counting.

This mechanism allows deterministic validation of AutoLISP libraries.

* IDE Capabilities
The development environment integrates with SLIME for interactive debugging.

Developers can inspect runtime state, step through code, and analyze stack frames.

The IDE environment also provides regression testing tools and snapshot diff utilities.

* Migration Toward Common Lisp
The interpreter allows gradual adoption of Common Lisp constructs.

Existing AutoLISP code can run unchanged while developers progressively adopt Common Lisp features.

This approach enables incremental modernization of legacy codebases.

* Roadmap
- Phase 1: Core interpreter and REPL.
- Phase 2: Visual LISP compatibility layer.
- Phase 3: Testing harness and snapshot system.
- Phase 4: CAD host integrations.
- Phase 5: UI subsystem and IDE tools.
