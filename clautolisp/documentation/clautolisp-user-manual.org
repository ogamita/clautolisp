* Scope
This manual documents the ~clautolisp~ standalone executable: how to
invoke it, what options it accepts, and what it does in each mode.

It does *not* document the AutoLISP / Visual LISP language. For the
language reference — surface syntax, builtins, semantics, and
host-product divergences — see the companion ~autolisp-spec~
subproject (sibling directory ~../../autolisp-spec/~).

This document must be kept in sync with the implementation. Whenever
~clautolisp/tools/clautolisp/source/main.lisp~ changes its option
parsing, banner, exit codes, or supported modes, this file must be
updated in the same commit.

The rendered PDF (~user-manual.pdf~, sibling of this file) is also
checked in. Whenever this Org file changes, run ~make user-manual~
(or ~make documentation~) to regenerate the PDF and stage both files
in the same commit.

* Synopsis
#+BEGIN_EXAMPLE
clautolisp [options] FILE.lsp
clautolisp [options] -x EXPRESSION
clautolisp [options]                 (interactive REPL)
#+END_EXAMPLE
#+TEXINFO_FILENAME: clautolisp-user-manual.info
#+TEXINFO_DIR_CATEGORY: clautolisp
#+TEXINFO_DIR_TITLE: clautolisp: (clautolisp-user-manual).
#+TEXINFO_DIR_DESC: Clautolisp standalone interpreter user manual.

The implementation is shipped as two image-built executables, one per
supported Common Lisp host:

- ~clautolisp-sbcl~ — built on SBCL via ~make build-clautolisp-sbcl~.
- ~clautolisp-ccl~  — built on CCL  via ~make build-clautolisp-ccl~.

Both binaries are produced under
~clautolisp/tools/clautolisp/bin/~ and accept the same command-line
surface. The rest of this manual treats them interchangeably and
refers to the binary as ~clautolisp~.

* Modes of Operation
~clautolisp~ has three mutually exclusive modes; the mode is
determined by the operands. In every mode the runtime installs the
core builtin function bindings before evaluation begins.

** File mode
A non-option argument is treated as a path to an AutoLISP source
file. The reader loads the file under the active dialect, the
evaluator processes every form sequentially, and the binary exits
once the file has been fully consumed.

#+BEGIN_EXAMPLE
clautolisp drawing-tools.lsp
clautolisp --bricscad path/to/script.lsp
#+END_EXAMPLE

** Expression mode
The ~-x EXPRESSION~ option evaluates the supplied string as if it
were the entire source of a file, then exits.

#+BEGIN_EXAMPLE
clautolisp -x "(setq x 7) (+ x 3)"
clautolisp --bricscad -x "(princ (atof \"0x1p4\"))"
#+END_EXAMPLE

The expression is read with the source name ~<-x>~, which appears in
diagnostic messages.

** Interactive REPL
With no ~FILE~ and no ~-x~, ~clautolisp~ drops into an interactive
read-eval-print loop on standard input.

- Primary prompt: ~_$ ~
- Continuation prompt (printed when an expression is incomplete): three spaces.
- Each balanced top-level form is evaluated and the result is printed
  using the AutoLISP value formatter (the same formatter used by
  ~prin1~).
- A reader or runtime error is reported as a one-line diagnostic; the
  REPL then re-prompts.
- Closing standard input (Ctrl-D on a fresh line) ends the REPL.

The REPL accepts multi-line input: as long as the parser reports
~:unexpected-eof~, the REPL keeps reading continuation lines and
appends them to the in-progress form before retrying.

An empty line just re-prompts.

*** REPL commands

A line whose first non-blank character is a comma is a /REPL command/,
not an expression (at a top-level read the comma is unambiguous —
AutoLISP has no backquote). Each command has a short key and a word
form:

| ~,d~ / ~,date~   | print the current date and time (ISO 8601)     |
| ~,u~ / ~,uptime~ | print how long this clautolisp process has run |
| ~,h~ / ~,help~   | list the REPL commands                         |
| ~,q~ / ~,quit~   | exit the REPL (useful when ~(quit)~ is not     |
|                  | available in the active dialect)               |

The commands live in a command dictionary of the REPL's /interactor/ —
the same mechanism behind the debugger's command dictionaries — so the
vocabulary can grow (and user commands can shadow it) uniformly across
all the interaction loops.

* Debugger

Clautolisp ships an interactive source-level debugger, *aldo*. It steps
through AutoLISP code, sets breakpoints on forms, inspects the call
stack and live values, and evaluates expressions in the context of a
stopped frame. The full command vocabulary is in the companion
/aldb Command Reference/ (=clautolisp-debugger-command-reference.org=);
this section is an orientation and a guide to the three user interfaces.

** Entering the debugger

Clautolisp is *always debugging*: the aldo thread is ready for the
lifetime of the process, so there is no "run" or "start" step. Control
reaches the debugger in one of three ways:

- *interrupt* — pressing =C-c= asks aldo to stop at the next instrumented
  form; a second =C-c= forces an immediate abort, skipping the debugger.
- *programmatic* — the AutoLISP forms ~(clal-break)~ and
  ~(clal-invoke-debugger [message])~ drop into aldo from your own code
  (the analogues of Common Lisp ~break~ / ~invoke-debugger~).
- *on error* — an uncaught AutoLISP error enters the debugger when the
  error policy asks for it. The policy is the ~--on-error
  quit|debug|ignore~ option (and the ~*clal-on-error*~ variable your
  code may rebind).

Only *instrumented* functions can be stepped into or stopped in;
instrument exactly what you want to debug (uninstrumented code runs at
full speed and is stepped over).

** The three user interfaces

aldo presents *one* command set through three interchangeable
interfaces; a feature available in one is available in all.

- *TUI* — the line-mode terminal interface. You type a short command
  token (or an AutoLISP form) and press =RET=. It needs nothing beyond a
  plain terminal, styles its output with ANSI colour where the terminal
  allows, and pages long output (see /Settings/ below). It is the
  fallback that always works, including inside an Emacs ~shell~ buffer.
- *ncurses* — a full-screen, four-pane interface (stack, source,
  interactor, REPL). Commands fire on a single keypress; you move the
  cursor onto a form and act on it (no line numbers to type). It runs on
  the terminal's alternate screen, so your program's display is restored
  when you resume.
- *aldb* — the Emacs interface (an SLDB-style minor mode). The debugger
  listens on a socket; from Emacs you connect with =M-x aldb-connect RET
  address:port RET=. Navigation and history reuse Emacs's own buffers and
  mark ring.

*Choosing the interface.* The default is the ~default-user-interface~
setting (=tui=, =ncurses= or =aldb=). A single run can override it on
the command line, without changing the saved default:

#+begin_example
  clautolisp --aldo-user-interface ncurses  program.lsp
  clautolisp --aldo-user-interface aldb \
             --aldb-listening-address 127.0.0.1 \
             --aldb-listening-port 4301          program.lsp
#+end_example

When aldb is selected and Emacs is not yet connected, aldo prints the
=M-x aldb-connect= line to the terminal and waits — you may instead pick
the TUI or ncurses there.

** Common commands

The same verbs work in every interface (typed as tokens in the TUI,
single keys in ncurses/aldb). The most used:

- Execution: ~continue~ (=c=), ~into~ (=i=, step into), ~next~ (=n=,
  step over), ~out~ (=o=, finish), ~quit~ (=q=, abort the run).
- Breakpoints: ~break~ (=b=) at a form / line; ~list breakpoints~ (=lb=);
  ~enable~ / ~disable~ / ~delete~ / ~condition~ / ~ignore~ a breakpoint.
- Stack: ~list frames~ (=lf=, the backtrace); the ~frame~ family —
  ~frame N~ (=f=), ~frame inner~/~outer~ (=fi=/=fo=),
  ~frame top~/~bottom~ (=ft=/=fb=); ~list locals~ / ~parameters~ /
  ~variables~ (=ll=/=lp=/=lv=).
- Data: ~print EXPR~ (=p=); typing a bare ~(form)~ evaluates it in the
  current frame; ~visit EXPR~ (=v=) opens the structured inspector.
- Source: ~list source~ (=ls=); ~help~ (=h= or =?=).

In ncurses and aldb you never type a line number: put the cursor on the
target form and the command takes the form at point.

** Navigating the source

The ~navigator~ setting chooses how each *stop* (a breakpoint, a step, an
error) presents itself:

- =sexp= (the default) — the debugger enters the *form navigator* directly:
  the source is shown with the current poll-point selected (~【…】~), the
  prompt is =NAV>=, and the motion keys below apply at once. =q= drops to
  the flat =DBG>= command loop.
- =line= — the flat presentation: a window of source lines around the
  current line (marked =>>=), at the =DBG>= prompt.

Beyond stepping, you can browse the source structurally — over one tree
that runs from a sub-expression out to the file system:

- =d= / =u= — descend into / ascend out of the selected node;
- =>= / =<= — next / previous sibling; =<<= / =>>= first / last;
- ~goto~ (=g=) — jump by name or path (=goto FILE:LINE=, =goto FN=, or a
  qualified path into nested local functions);
- ~definition~ (=.=) — open the definition of the selected call;
  ~back~ / ~history~ — return through where you have been.

Browsing never changes what executes; ~print~, ~setq~ and friends always
act in the current frame.

In the navigator every debugger command is also available directly (the
navigator's dictionary is stacked over the debugger's); ~aldo CMD~ /
~debug CMD~ force the debugger's meaning of a command the navigator
shadows (e.g. ~aldo b LOC~, while bare =b= breakpoints the selection) —
the same prefix the sedit editor uses.

** Settings and configuration

Per-session options are read and set from any interface:

- =,settings= — list every option and its value; =,settings NAME= —
  print one.
- =set NAME VALUE= — set one (names and values are case-insensitive).
  Common options: ~navigator~ (=sexp= | =line=), ~theme~ (=unicode= |
  =ascii= | =white-on-black= | =black-on-white= | =green-on-black= |
  =attributes=), ~pager~ (=on= | =off=), ~pager-height~,
  ~source-window-height~, ~value-line-width~, ~break-on-caught~.

The configuration lives in an AutoLISP sexp bound to
~*clal-aldo-configuration*~, read at start-up from
=$XDG_CONFIG_HOME/clautolisp/aldo.conf= (or searched along
=$XDG_CONFIG_DIRS=) *before* your rc/init files, which may then modify
it. Changes made with =set= update the variable in memory only; nothing
is written automatically. Save and reload explicitly:

- =,settings save= — write the configuration to
  =$XDG_CONFIG_HOME/clautolisp/aldo.conf= (UTF-8);
- =,settings reload= — re-read it.

The same is available from AutoLISP as ~(clal-save-aldo-configuration)~
and ~(clal-load-aldo-configuration)~.

** Long output and the pager

Commands whose output exceeds the screen — =help=, a long
=list source=, a big value — are shown a page at a time in the TUI when
~pager~ is =on= (the default on a size-limited terminal; you typically
turn it =off= under an Emacs ~shell~, which scrolls). The pager accepts
the familiar =less= keys (=SPACE=/=f= forward, =b= back, =q= quit). The
ncurses interface uses a scrollable window instead, and aldb relies on
Emacs's own buffer scrolling.

* sedit and the aldo form navigator

Clautolisp includes *sedit*, an S-expression *structural* editor: you edit
a program as a tree of forms rather than as characters. It is the same tool
in two guises — a standalone editor you call from AutoLISP with
~(clal-sedit …)~, and the *form navigator* the aldo debugger opens over live
code. This section is an orientation; the normative specification is
=clautolisp/documentation/clautolisp-sedit-spec.org=.

** Two modes: navigate and edit

sedit is always in one of two modes.

- *Navigate* (NAV) — the selection moves over the *evaluable* sub-forms of a
  program (the same poll-points the debugger stops at); structure is not
  changed. This is the mode the debugger uses; its keys are the ones in
  /Debugger → Navigating the source/ above (=d= =u= descend/ascend, =>= =<=
  siblings, =<<= =>>= first/last, =±N= skip).
- *Edit* (EDIT) — the selection moves over *every* sub-form and comment, and
  the full editing command set is available.

The selection is shown in brackets, e.g. ~(list [nil])~. Motion and mode
switching:

| Key / word          | Effect                                                    |
|---------------------+-----------------------------------------------------------|
| =d= =u=             | descend into / ascend out of the selection                |
| =>= =<= (=f= =b=)    | next / previous sibling (=<<= =>>= first / last, =±N= skip)|
| ~edit~              | switch to EDIT (an editing command by *word* also does)   |
| ~nav~               | switch back to NAV                                         |
| ~debug CMD~ / ~aldo CMD~ | from EDIT, run a debugger command CMD, then return to NAV |
| =q=                 | leave the editor, returning the edited object             |
| =h= / =?=           | help                                                      |

Because single letters mean different things in the debugger and the editor,
an editing command is recognised in NAV *by its word* (~insert~, ~wrap~, …):
typing one enters EDIT and performs it.

** Editing commands (EDIT)

- *Insert / replace* — ~insert σ~ (before the selection), ~add σ~ (after),
  ~replace σ~ (in place); each reads the new form σ after the command and
  selects it. Comment variants ~ic~ / ~ac~ / ~rc~ attach a comment.
- *Structural* — ~wrap~ (put the selection in a new list), ~splice~ (its
  inverse), ~slurp~ / ~barf~ (pull the next sibling in / push the last out),
  ~split~ / ~join~.
- *Clipboard* — ~copy~ (=c=), ~cut~ (=x=), ~paste~ (=v=); see /The clipboard/.
- *Undo* — ~undo~ (=z=): one level; a second ~undo~ redoes.
- *Evaluate* — ~eval~ (=e=) evaluates the selection; ~macroexpand~ (=m=)
  replaces it by the result. Typing a bare Lisp *form* (~(+ 1 2)~) at the
  prompt evaluates it and prints the result, like the REPL — in every mode
  (=DBG>=, =NAV>= and =SEDIT>=).
- *Files & directories* — ~load~ (=l=) and ~save~ (=s=); in a directory,
  ~new~ (=n=), ~rename~ (=r=) and ~delete~ (=x=).

sedit preserves *formatting*: an unmodified form is written back byte-for-byte
(comments and layout intact), and a form you edit is re-indented by AutoLISP's
own rules.

** Starting sedit from AutoLISP

#+begin_example
(clal-sedit)                 ; a fresh stand-alone form, starting at nil
(clal-sedit 'foo)            ; recall and edit the definition of FOO
(clal-sedit "prog.lsp")      ; edit a file (its first top-level form selected)
(clal-sedit "~/src/")        ; browse a directory
(clal-sedit '(+ 1 2))        ; edit a value as a sexp
#+end_example

~clal-sedit~ runs the editor on the terminal and returns the edited object.
Recall by name works because the interactive REPL *records* every top-level
form you evaluate, so ~(clal-sedit 'foo)~ finds the ~defun~ / ~setq~ you
typed. The initial and final forms are left in ~*clal-sedit-initial-form*~ and
~*clal-sedit-last-result*~.

** Editing while debugging

From the aldo form navigator, issuing an editing word (~edit~, ~insert~,
~wrap~, …) opens sedit on the *current function's* source. Edit it, then
leave the editor (=q=): the new definition is *installed* in the running
system, so the next call runs the edited code. Breakpoints set by poll-point
identity carry across the redefinition. While editing, ~debug CMD~ / ~aldo
CMD~ still reach the debugger, and ~nav~ returns to plain navigation.

** The clipboard

sedit keeps *two* clipboards in sync: an in-process one (full structural
fidelity, the source of truth) and the *system* clipboard (a textual
rendering other applications see). Copy/cut write both; paste reads the
system clipboard and parses it back into a form, falling back to the
in-process clipboard when the system one is empty. The AutoLISP surface:

| Function                          | Effect                                        |
|-----------------------------------+-----------------------------------------------|
| ~(clal-clipboard-copy-sexp OBJ)~  | copy OBJ (internal + system as source text)   |
| ~(clal-clipboard-paste-sexp)~     | the clipboard parsed back to a form, or ~nil~ |
| ~(clal-clipboard-put-text S)~     | set the system clipboard to string S          |
| ~(clal-clipboard-get-text)~       | the system clipboard as a string, or ~nil~    |

Paste *never evaluates* its input (a foreign ~#.(…)~ on the clipboard cannot
run code). The system clipboard is reached through a provider that
auto-selects among X11, Wayland, macOS, Windows and OSC 52, and degrades to
in-process only when none is available — clautolisp needs no GUI-toolkit
dependency. Providers, serialization and security are specified in
=clautolisp/documentation/clipboard-interface.org=. Toggle the system
mirroring off by binding ~*system-clipboard-enabled*~ to ~nil~.

* Options
All options begin with two dashes except for ~-x~. They may appear in
any order before, after, or interleaved with the ~FILE~ argument; the
last occurrence of an option wins. ~--~ is *not* a recognised
end-of-options marker; the runtime forwards every CLI argument to the
parser.

** Dialect selection
The dialect descriptor controls reader strictness, host-specific
permissive parsing, and the runtime knobs documented in
~autolisp-spec~. The default is the conservative ~strict~ profile.

- ~--dialect NAME~       :: Select the named dialect. NAME is one of
                            ~strict~, ~autocad-2026~, ~bricscad-v26~,
                            ~clautolisp~, ~lax~.
- ~--strict~             :: Equivalent to ~--dialect strict~.
- ~--autocad~            :: Equivalent to ~--dialect autocad-2026~.
- ~--bricscad~           :: Equivalent to ~--dialect bricscad-v26~.
- ~--clautolisp~         :: Equivalent to ~--dialect clautolisp~.
- ~--lax~                :: Equivalent to ~--dialect lax~. Silences
                            every encoding-dispatch diagnostic
                            globally; useful when consuming code from
                            multiple vendors. The runtime still
                            honours each form; only the diagnostic
                            is suppressed.

The ~clautolisp~ dialect includes `clautolisp extensions`
(eg. variadic functions).

Whatever the dialect selected, all the operators are available. Only
those that are not specifically inside the set defined by the dialect
produce warnings when compiled and when used.

Unknown dialect names produce a diagnostic on standard error and exit
code 1.

** Input selection
- ~-x EXPRESSION~        :: Evaluate EXPRESSION instead of reading a
                            file. Implies expression mode. Must be
                            followed by exactly one argument.

** REPL behaviour
- ~--quiet~              :: Suppress the REPL banner; only the prompt
                            is shown. No effect outside REPL mode.

** Host backend
The Host Abstraction Layer (HAL, Phase 8 of the implementation
roadmap) decides where CAD-host effects land. The dialect descriptor
controls the *language*; the host backend controls the *target*.
Both axes are independent.

- ~--host NAME~          :: Select the backend. Currently accepted:
                            ~mock~ (default since 0.0.14): an
                            in-memory deterministic CAD-database
                            substitute capable of entget / ssget /
                            getvar / getstring etc. ~null~: every
                            CAD-host operation signals
                            ~:host-not-supported~ — useful for
                            programs that should not touch a host
                            at all. ~live~ is reserved for Phase 16.
- ~--mock-input PATH~    :: Attach the file at PATH as the
                            MockHost's prompt-stream. Lines are
                            consumed in order by ~getstring~,
                            ~getreal~, ~getint~, ~getpoint~, etc.
                            so headless test rigs can feed
                            deterministic answers to interactive
                            scripts.

** Colour output
~clautolisp~ accents AutoLISP-symbol names in error messages,
backtraces, and trace output with an ANSI colour when standard
output is a terminal. The default is conservative (yellow on
assumed-dark) and never fires when the output is being captured
(pipe, file redirect, test harness).

- ~--no-color~           :: Disable colour for this invocation.
                            Equivalent to setting ~$NO_COLOR~ to
                            any non-empty value (see https://no-color.org).
- ~$NO_COLOR~            :: Universal no-color convention; honoured
                            equivalently to ~--no-color~.
- ~$CLAUTOLISP_SYMBOL_FOREGROUND=NAME~ ::
                            Force the foreground colour for printed
                            AutoLISP symbols. Accepted names (case
                            and whitespace insensitive): ~black~,
                            ~red~, ~green~, ~yellow~, ~blue~,
                            ~magenta~, ~cyan~, ~white~ and their
                            ~bright-~ prefixed variants
                            (~bright-black~ is also ~grey~ / ~gray~).
                            ~default~ / ~none~ / ~off~ leaves the
                            foreground axis bare.
- ~$CLAUTOLISP_SYMBOL_BACKGROUND=NAME~ ::
                            Force the background colour for printed
                            AutoLISP symbols. Same vocabulary as the
                            foreground sibling; either or both env
                            vars may be set. Both win over the
                            automatic luminance probe so the user
                            gets a deterministic look without
                            depending on terminal detection.
- ~$CLAUTOLISP_BACKGROUND=dark|light~ ::
                            Override the terminal-background guess
                            without specifying a colour. ~dark~ keeps
                            the yellow accent; ~light~ flips it to
                            blue.
- ~$CLAUTOLISP_PROBE_OSC11~ ::
                            Opt-in to the active OSC 11 terminal
                            query (spawns ~/bin/sh~ + ~stty~ + ~dd~
                            against ~/dev/tty~). Off by default
                            because on some platforms — notably
                            macOS arm64 — the helper subprocess
                            receives SIGTTOU and hangs CLI startup.
                            Safe to set on most Linux desktops and
                            iTerm2 / Terminal.app launched from a
                            normal shell.

Example: yellow symbols on a dim-blue background, deterministically,
no probing:

#+BEGIN_SRC sh
  export CLAUTOLISP_SYMBOL_FOREGROUND=bright-yellow
  export CLAUTOLISP_SYMBOL_BACKGROUND=blue
  clautolisp -x '(foreach x t (princ x))'
#+END_SRC

** Source-file and terminal encoding
- ~-e ENC~                :: Set the default source-file encoding for
                             every ~load~ in the session, including
                             nested ~(load …)~ calls from init files.
                             ENC is one of ~utf-8~, ~iso-8859-1~,
                             ~latin-1~, ~windows-1252~, ~cp1252~.
                             Overrides the dialect default and the
                             POSIX-locale guess (~LC_ALL~ /
                             ~LC_CTYPE~ / ~LANG~). Surfaced to user
                             code as the ~*AUTOLISP-FILE-ENCODING*~
                             global.
- ~-E ENC~                :: Declared terminal-IO encoding for the
                             session. Surfaced to user code as the
                             ~*AUTOLISP-TERMINAL-ENCODING*~ global so
                             code that emits raw bytes can adapt. No
                             stream rebinding is performed at the CL
                             level — set this when your downstream
                             tooling needs the declared encoding to
                             match what you actually emit. Same ENC
                             vocabulary as ~-e~.

*** Launch-chain resolution and the degenerate case
At startup the runtime resolves the file encoding via
~or-else(LC_ALL, LANG, LC_CTYPE, -e flag)~ and pushes the result
into two slots: ~SYSCODEPAGE~ (read-only host sysvar) and
~*AUTOLISP-FILE-ENCODING*~ (read-write global the user may
~setq~ at the REPL or in init files).

When ~-e~ is absent AND all three locale env vars are unset or
empty (the *degenerate case* — common in CI containers and
freshly-provisioned cloud hosts), the two slots diverge:

- ~SYSCODEPAGE~ / ~DWGCODEPAGE~ -> ~"US-ASCII"~. The safest 7-bit
  assumption a vendor-compatible host sysvar can carry. User code
  that branches on ~(getvar "SYSCODEPAGE")~ gets a non-empty
  deterministic value.

- ~*AUTOLISP-FILE-ENCODING*~ -> ~""~ (the **empty string**). The
  global is bound to a non-nil autolisp-string, but the empty
  value is a sentinel meaning "no preference — LOAD/OPEN should
  consult the dialect default." The runtime's
  ~LOOKUP-AUTOLISP-FILE-ENCODING~ helper treats empty the same
  way it treats an unbound global, so the LOAD precedence chain
  falls through to the dialect default:
  ~--clautolisp~ / ~--autocad-2026~ / ~--bricscad-v26~ -> UTF-8;
  ~--strict~ -> ISO-8859-1.

Observable from AutoLISP under the degenerate case:

#+BEGIN_SRC autolisp
*AUTOLISP-FILE-ENCODING*           ; => ""           (bound, non-nil)
(strlen *AUTOLISP-FILE-ENCODING*)  ; => 0
(getvar "SYSCODEPAGE")             ; => "US-ASCII"
(load "lib/has-utf8-bytes")        ; loaded with UTF-8 under --clautolisp
#+END_SRC

Why the asymmetry: pinning ~*AUTOLISP-FILE-ENCODING*~ to
~"US-ASCII"~ in the degenerate case would make every source-file
~load~ strict ASCII regardless of dialect, and the first
non-ASCII byte in any init file raises a stream decoding error.
A 1-to-1 8-bit fallback (~ISO-8859-1~) would avoid the crash —
every byte is a valid ISO-8859-1 character — but at the cost of
silently mojibake'ing UTF-8 multi-byte sequences into pairs of
Latin-1 characters. The empty-string sentinel cleanly hands the
decision back to the dialect, which already declares what
encoding its source files use.

The full design rationale is in
~issues/closed/encoding-dispatch.issue~ §"Degenerate launch case."

** Informational
- ~--version~            :: Print =clautolisp <version>= on standard
                            output and exit 0. The version is the
                            value of ~clautolisp.tools.clautolisp:*version*~
                            (format ~MAJOR.MINOR.DEVELOP~).
- ~--help~               :: Print a short usage summary and exit 0.

* Image-introspection variables (~*AUTOLISP-…*~)
At startup ~clautolisp~ publishes a set of globally-bound variables
in the running AutoLISP image describing how it was invoked. User
code can read them to branch on backend, dialect, encoding, etc.
See ~issues/closed/transmit-options.issue~ for the full design and
the alfe-side counterpart.

** Static — bound once, before the first user action
- ~*AUTOLISP-FRONTEND*~         :: The symbol ~CLAUTOLISP~ — clautolisp
                                   was the entry point. (Under alfe
                                   this is ~ALFE~.)
- ~*AUTOLISP-BACKEND*~          :: The symbol ~CLAUTOLISP~ — the
                                   engine running this code.
- ~*AUTOLISP-DIALECT*~          :: The active dialect's canonical
                                   name as a symbol — ~STRICT~,
                                   ~AUTOCAD-2026~, ~BRICSCAD-V26~.
- ~*AUTOLISP-HOST*~             :: The HAL backend's name as a symbol
                                   — ~MOCK-HOST~, ~NULL-HOST~, or nil.
- ~*AUTOLISP-MOCK-INPUT*~       :: Pathname (string) passed to
                                   ~--mock-input~, or nil.
- ~*AUTOLISP-GUI-COMMAND*~      :: Subprocess renderer command
                                   passed to ~--gui~, or nil.
- ~*AUTOLISP-TRACE*~            :: T when ~--trace~ was on at startup.
- ~*AUTOLISP-FILE-ENCODING*~    :: Always a string (never nil).
                                   Resolved at launch from
                                   ~-e ENC~ (explicit), then the
                                   POSIX locale chain
                                   (~LC_ALL~ > ~LANG~ >
                                   ~LC_CTYPE~), falling back to
                                   the empty string ~""~ when
                                   nothing resolves. The empty
                                   value is a sentinel — LOAD /
                                   OPEN treat it as "no preference"
                                   and fall through to the dialect
                                   default (UTF-8 for non-strict
                                   dialects, ISO-8859-1 for
                                   ~--strict~). See "Launch-chain
                                   resolution and the degenerate
                                   case" under *Options*.
- ~*AUTOLISP-TERMINAL-ENCODING*~ :: Always a string. Same
                                    resolution chain as
                                    ~*AUTOLISP-FILE-ENCODING*~,
                                    but falls back to ~"US-ASCII"~
                                    rather than the empty string —
                                    terminal IO does not have a
                                    "dialect default" to fall
                                    through to.
- ~*AUTOLISP-NO-INIT*~          :: T when ~-norc~ / ~--no-init~ was
                                   passed, or the env-var override
                                   suppressed init files.
- ~*AUTOLISP-NO-COLOR*~         :: T when ~--no-color~ was passed.
- ~*AUTOLISP-VERBOSE*~          :: T when ~-v~ / ~--verbose~ was
                                   passed.
- ~*AUTOLISP-DEBUG*~            :: T when ~-d~ / ~--debug~ was passed.
- ~*AUTOLISP-QUIET*~            :: T when ~-q~ / ~--quiet~ was passed.
- ~*AUTOLISP-WILL-QUIT*~        :: Reserved for the ~--quit~ flag;
                                   always nil under clautolisp
                                   today. alfe sets it.
- ~*AUTOLISP-MAIN*~             :: Reserved for the ~--main~ flag;
                                   always nil under clautolisp
                                   today.
- ~*AUTOLISP-ACTIONS*~          :: Ordered list of the actions parsed
                                   off the command line, each shaped
                                   ~(load "PATH")~, ~(eval "TEXT")~, or
                                   ~(interactive T)~. Empty list when
                                   the implicit REPL is the only
                                   action.
- ~*AUTOLISP-HELP*~             :: String containing the ~--help~
                                   output.
- ~*AUTOLISP-VERSION*~          :: String like ~"1.0.47"~.

User code may ~setq~ any of these; the runtime does not re-read the
values, so the modification is purely an information change visible
to subsequent ~boundp~/value lookups.

** Dynamic — bound only while the matching action runs
- ~*AUTOLISP-LOAD-PATHNAME*~    :: The pathname of the file being
                                   loaded by a ~-l~ action. Bound to
                                   the path string for the duration
                                   of the ~load~ call; reset to nil
                                   on return (success or signal).
- ~*AUTOLISP-EXPRESSION*~       :: The source text of a ~-x~ action,
                                   bound while the expression is
                                   evaluated. Cleared on return.
- ~*AUTOLISP-INTERACTIVE*~      :: T while the REPL is active.
                                   Cleared on REPL exit.

Code running inside an init file can therefore distinguish "loaded
because of ~-l init.lsp~" (~*AUTOLISP-LOAD-PATHNAME*~ points at
~init.lsp~) from "loaded as an init file from the standard search
path" (~*AUTOLISP-LOAD-PATHNAME*~ is nil).

* Banner
In REPL mode (and only in REPL mode) ~clautolisp~ prints a one-line
banner before the first prompt:

#+BEGIN_EXAMPLE
clautolisp 0.0.9 — REPL (STRICT dialect, null-host host) — Ctrl-D to exit.
#+END_EXAMPLE

The version string, dialect name, and host backend name reflect the
current build and the selected ~--dialect~ / ~--host~. Pass
~--quiet~ to suppress the banner.

* Exit Codes
~clautolisp~ uses the following exit codes. They are stable and may
be relied upon by shell scripts.

- ~0~  :: Successful completion (file or ~-x~ ran without error;
          REPL exited cleanly via end-of-input; ~--help~ /
          ~--version~ printed and exited; an AutoLISP ~exit~ /
          ~quit~ termination was raised cleanly).
- ~1~  :: A runtime error escaped the AutoLISP error-handling layer,
          or the option parser rejected an unknown flag.
- ~2~  :: The requested file could not be opened (caught Common-Lisp
          ~file-error~).

Within the REPL, individual evaluation errors are reported but do
*not* terminate the loop; the binary will still exit 0 once stdin
closes.

* Errors and Diagnostics
- Reader errors are produced by the AutoLISP reader and surface as
  ~simple-error~ conditions whose printable form embeds a structured
  ~diagnostic~ (severity, code, message, source span). They are
  printed to standard error in batch modes and to standard error in
  REPL mode while the loop continues.
- Runtime errors are signalled as ~autolisp-runtime-error~ with a
  symbolic ~code~ (e.g. ~UNDEFINED-FUNCTION~, ~UNBOUND-VARIABLE~,
  ~TYPE-ERROR~) and a human-readable message.
- Termination conditions raised by ~exit~ / ~quit~ (~autolisp-termination~)
  surface as a one-line notice on standard error and exit cleanly.

The exact set of error codes lives in the runtime; see
~autolisp-runtime/source/api.lisp~ for the canonical mapping.

** Encoding-dispatch diagnostics (~enc-*~)
Each dialect dispatches a distinct diagnostic when user code uses an
encoding-related extension that belongs to a different dialect. The
diagnostics are *advisory*: the runtime always honours the form so
user code stays runnable across dialects; the diagnostic just
surfaces the portability concern.

Eight codes, all prefixed ~enc-~:

| Code                        | When                                             |
|-----------------------------+--------------------------------------------------|
| ~enc-foreign-dialect~       | a form belonging to a non-selected dialect       |
| ~enc-extension-used~        | ~--strict~ only: any encoding extension          |
| ~enc-unsupported-target~    | selected encoding has no host expression         |
| ~enc-invalid-value~         | bad encoding name string                         |
| ~enc-lispsys-out-of-range~  | ~(setvar "LISPSYS" n)~ with ~n \notin {0,1,2}~   |
| ~enc-codepage-mismatch~     | ~DWGCODEPAGE~ differs from ~SYSCODEPAGE~         |
| ~enc-unknown-codepage~      | unknown ~SYSCODEPAGE~ string                     |
| ~enc-host-dependent~        | ~"ANSI"~ used in a write context (info)          |

Diagnostics print to standard error as
~[enc-<code>] <message>~. The output stream is overridable for
tests via the CL parameter
~clautolisp.autolisp-runtime:*enc-diagnostic-stream*~; emission is
silenced entirely by binding
~*enc-diagnostic-suppress-p*~ non-nil (the runtime side-door for
the per-file pragma mechanism described below).

*Two user-facing suppression knobs:*

- *Per-code pragma.* Inside a source file:

  #+BEGIN_SRC autolisp
  (clal-suppress-enc-diagnostic 'enc-extension-used 'enc-foreign-dialect)
  ;; remainder of the file emits nothing for the two listed codes
  #+END_SRC

  Re-enable via ~(clal-enable-enc-diagnostic 'enc-extension-used)~
  or, without arguments, ~(clal-enable-enc-diagnostic)~ to clear
  the entire suppression set. Code names accept either a string
  (~"enc-extension-used"~) or a symbol (~'enc-extension-used~);
  the leading ~enc-~ is optional.

- *--lax dialect.* Run clautolisp with ~--lax~ (or
  ~--dialect lax~) to silence *every* encoding diagnostic
  globally for the session. Useful when consuming code from
  multiple vendors and you don't want to commit to one
  dialect's portability rules. The runtime still honours each
  encoding form; only the diagnostic is suppressed.

*Static encoding-extension linter*:

~(clal-lint-encoding-extensions FORM)~ walks FORM recursively
and emits the same enc-* diagnostics the runtime would, without
actually evaluating any call. Useful for catching dead-code
encoding extensions before they ship — pipe a file's parsed
form-tree through the lint at compile time:

#+BEGIN_SRC autolisp
;; Hypothetical CI gate:
(clal-lint-encoding-extensions
  (read-from-string-or-something-like-it
   (slurp "lib/encoding-aware.lsp")))
;; -> [enc-extension-used] LOAD's positional [encoding] argument...
;;    [enc-foreign-dialect] OPEN CCS-SUFFIX ...
#+END_SRC

The lint shares SIGNAL-ENCODING-DIAGNOSTIC with the runtime
emitters, so dialect / pragma / --lax gates apply identically.
Returns ~T~ when the walk completes.

Currently wired:

- ~LOAD~ with the positional ~[encoding]~ argument under
  ~--strict~ (~enc-extension-used~), ~--autocad~ /
  ~--bricscad~ (~enc-foreign-dialect~), silent under
  ~--clautolisp~.
- ~OPEN~ encoding forms, all three routed:
  - positional ~"utf8"~ / ~"utf8-bom"~ — native under
    ~--autocad~; foreign under ~--bricscad~ / ~--clautolisp~;
    extension-used under ~--strict~.
  - positional ~"UTF-8"~ / ~"UTF-8-BOM"~ / ~"CP-NNNN"~ / … —
    native under ~--clautolisp~; foreign under ~--autocad~ /
    ~--bricscad~; extension-used under ~--strict~.
  - ~,ccs=ENC~ mode-suffix — native under ~--bricscad~;
    foreign under ~--autocad~ / ~--clautolisp~; extension-used
    under ~--strict~. The suffix is stripped from the
    mode-string before CL ~OPEN~ sees it.
- ~GETVAR~ / ~SETVAR~ on ~LISPSYS~ under every dialect except
  ~--autocad~ (the native dialect of ~LISPSYS~):
  - ~--strict~ -> ~enc-extension-used~.
  - ~--bricscad~ / ~--clautolisp~ -> ~enc-foreign-dialect~ with
    the dialect sub-tag.
- ~SETVAR "LISPSYS" <n>~ with ~n \notin {0,1,2}~ emits
  ~enc-lispsys-out-of-range~ regardless of dialect (the spec mandates
  the domain). The write proceeds — matching the vendor "permissive
  but warn" behaviour.
- ~OPEN~ with an encoding the active host can't natively express
  (e.g. ~"UTF-16-LE"~ under ~--autocad~) emits
  ~enc-unsupported-target~. Informational; the runtime still
  attempts the open.
- ~OPEN~ in WRITE direction with ~"ANSI"~ / ~"MBCS"~ emits
  ~enc-host-dependent~ (info-level): output resolves through
  the host's SYSCODEPAGE at I/O time and isn't portable across
  hosts. Reading under those encodings is silent.
- The internal SET-DRAWING-CODEPAGE wrapper (CL-level; hook for
  future drawing-load paths) emits ~enc-codepage-mismatch~ when
  the incoming DWGCODEPAGE canonical form differs from
  SYSCODEPAGE — catches the "Czech-authored drawing on a French
  host" failure mode before string decode time.
- ~%CANONICAL-CODEPAGE-STRING~ emits ~enc-unknown-codepage~ when
  asked to canonicalise a spelling not in the known table (typos,
  unsupported vendor aliases). The raw value passes through.

The full set of codes, the per-dialect dispatch table, and the
roadmap for the remaining hooks (~OPEN~, BricsCAD ~,ccs=~ parsing,
~DWGCODEPAGE~ mismatch detection at drawing-load time) is in
~issues/closed/encoding-dispatch.issue~.

$ clautolisp --version

#+END_EXAMPLE

* Building the Executable
From the ~clautolisp/~ subproject root:

#+BEGIN_EXAMPLE
make build-clautolisp-sbcl   # produces tools/clautolisp/bin/clautolisp-sbcl
make build-clautolisp-ccl    # produces tools/clautolisp/bin/clautolisp-ccl
#+END_EXAMPLE

Both targets reuse the ~read-autolisp~ build harness. The SBCL build
sets ~:save-runtime-options t~ so the saved core forwards every
command-line argument to ~clautolisp~ rather than letting SBCL
intercept ~--help~ or other runtime flags.

* Versioning
The version string is stored in
~clautolisp/tools/clautolisp/source/version.lisp~ as
~*version*~. The format is ~MAJOR.MINOR.DEVELOP~. The DEVELOP
counter is bumped on every change that touches clautolisp source
code; the running binary's ~--version~ output is therefore a stable
identifier of the build.

* clautolisp Extensions
The core builtin registry installed by ~clautolisp~ is a *superset*
of the operators listed in the local AutoLISP / Visual LISP
specification draft. The extras documented in this section are
convenience extensions and compatibility helpers that real-world
AutoLISP code relies on without going through the spec process.

Code that targets the strict shared standard (the AutoCAD/BricsCAD
intersection) MUST NOT depend on the operators below; tests
written against ~autolisp-test/harness/inventory.sexp~ are
intentionally restricted to the spec, and the inventory does not
list these names.

** Compositional list accessors (~Cxxxr~ / ~Cxxxxr~ family)
The local specification documents ~CAR~, ~CDR~ and ~CADR~. Beyond
those, ~clautolisp~ also installs:

- two-level :: ~CAAR~, ~CDAR~, ~CDDR~
- three-level :: ~CAAAR~, ~CAADR~, ~CADAR~, ~CADDR~, ~CDAAR~,
  ~CDADR~, ~CDDAR~, ~CDDDR~
- four-level :: ~CAAAAR~, ~CAAADR~, ~CAADAR~, ~CAADDR~, ~CADAAR~,
  ~CADADR~, ~CADDAR~, ~CADDDR~, ~CDAAAR~, ~CDAADR~, ~CDADAR~,
  ~CDADDR~, ~CDDAAR~, ~CDDADR~, ~CDDDAR~, ~CDDDDR~

Each composition delegates to ~CAR~ / ~CDR~ so a non-list argument
anywhere along the chain still surfaces a CAR-shaped or CDR-shaped
runtime error under the proper operator name.

#+BEGIN_SRC autolisp
$ clautolisp -x "(cadddr '(a b c d e))"
D
#+END_SRC

** ~ERROR~
~(error MESSAGE-OR-STRING ...)~ raises an AutoLISP-visible
runtime error with code ~:user-error~ and the supplied message.
Modeled after the common Visual LISP idiom of calling ~(error ...)~
to abort with a user-facing message, and catchable with
~vl-catch-all-apply~.

#+BEGIN_SRC autolisp
$ clautolisp -x "(vl-catch-all-error-message
                   (vl-catch-all-apply 'error '(\"oops\")))"
"oops"
#+END_SRC

** ~LCM~, ~LOGXOR~
Convenience numerics that complement the in-spec ~GCD~, ~LOGAND~ and
~LOGIOR~. ~LCM~ takes two or more integers and returns their least
common multiple; ~LOGXOR~ takes two or more integers and returns
their bitwise exclusive-or.

#+BEGIN_SRC autolisp
$ clautolisp -x "(lcm 4 6)"
12
$ clautolisp -x "(logxor 12 10)"
6
#+END_SRC

** ~VL-BT~ — full backtrace
The local specification reserves ~VL-BT~ as a Visual LISP backtrace
operator. ~clautolisp~ implements it as a real backtrace printer
rather than a stub: ~(vl-bt)~ prints the AutoLISP call stack at the
current evaluation point, one line per frame, most recent first.

The runtime tracks the stack through ~*autolisp-call-stack*~, a
dynamically-bound list of evaluator frames pushed by ~autolisp-eval~
and by every SUBR / USUBR call. Frame kinds are ~:eval~,
~:special-op~, ~:subr~ and ~:usubr~.

** ~VL-CATCH-ALL-ERROR-STACK~ — captured backtrace
~(vl-catch-all-error-stack OBJECT)~ is a clautolisp-only extension
companion to the standard ~vl-catch-all-error-message~. Given a
catch-all error object, it returns the AutoLISP call-stack snapshot
captured at the time the underlying ~autolisp-runtime-error~ was
raised. Each frame is a cons ~(KIND . PAYLOAD)~, where ~PAYLOAD~ is
either the form being evaluated (for ~:eval~ / ~:special-op~) or
the cons ~(NAME . ARGS)~ of the function call (for ~:subr~ /
~:usubr~).

This is the operator the ~autolisp-test~ harness calls to render an
AutoLISP backtrace inside a FAIL detail when a tested function
raises. AutoCAD and BricsCAD do not expose the captured stack, so
the harness gracefully falls back to message-only reporting on those
hosts.

#+BEGIN_SRC autolisp
$ clautolisp -x "(setq c (vl-catch-all-apply 'car '(7)))
                 (length (vl-catch-all-error-stack c))"
;; one or more frames captured at the time CAR was rejected
3
#+END_SRC

Code that needs to remain portable should test for the operator
before calling it:

#+BEGIN_SRC autolisp
(if (boundp 'vl-catch-all-error-stack)
    (vl-catch-all-error-stack object)
    nil)
#+END_SRC

** The ~CLAL-~ namespace (clautolisp-only helpers)
The four other sub-sections above (~Cxxxr~ accessors, ~ERROR~,
~LCM~/~LOGXOR~, ~VL-BT~/~VL-CATCH-ALL-ERROR-STACK~) cover
extensions that complete an established AutoCAD/BricsCAD idiom
without inventing a new one. From clautolisp 1.0.76 onward, a
second class of extensions ships under a reserved ~CLAL-~ prefix:
helpers that have *no* vendor counterpart and that we want to make
trivially distinguishable from any future Autodesk- or Bricsys-
documented name. Code that calls one of these functions is
explicitly opting into the clautolisp dialect.

The probing convention for portable code:

#+BEGIN_SRC autolisp
(if (boundp 'clal-sysvar-list)
    (clal-sysvar-list)
    ;; vendor fallback -- there isn't one; either accept that the
    ;; query is unavailable or fall back to the SETVAR command.
    nil)
#+END_SRC

*** ~CLAL-SYSVAR-LIST~ -- list every sysvar known to the host
~(clal-sysvar-list)~ returns a list of strings: the sysvar names
the active host (mock or live) reports, sorted in case-sensitive
ASCII order. On a default ~MockHost~ the list has 1836 entries --
the same universe as the autolisp-spec §16 *System Variables*
catalogue.

#+BEGIN_SRC autolisp
$ clautolisp -x "(length (clal-sysvar-list))"
1836

$ clautolisp -x "(car (clal-sysvar-list))"
"3DCONVERSIONMODE"
#+END_SRC

*** ~CLAL-SYSVAR-APROPOS~ -- substring search over sysvar names
~(clal-sysvar-apropos PATTERN)~ returns the subset of
~(clal-sysvar-list)~ whose names contain ~PATTERN~ as a
case-insensitive substring. Useful at the REPL when you remember
"there's something with ~ANG~ in it" but not the exact name.

#+BEGIN_SRC autolisp
$ clautolisp -x "(clal-sysvar-apropos \"ANG\")"
("ANGBASE" "ANGDIR" "INPUTHISTORYMODE" ...)

$ clautolisp -x "(clal-sysvar-apropos \"ang\")"     ;; case-insensitive
("ANGBASE" "ANGDIR" "INPUTHISTORYMODE" ...)

$ clautolisp -x "(clal-sysvar-apropos \"ERRNO\")"
("ERRNO")

$ clautolisp -x "(length (clal-sysvar-apropos \"\"))"   ;; empty -> all
1836
#+END_SRC

A non-string PATTERN raises ~:invalid-argument~. A pattern that
matches nothing returns ~nil~.

*** ~CLAL-SYSTEM-CODEPAGE~ -- canonical SYSCODEPAGE
~(clal-system-codepage)~ wraps ~(getvar "SYSCODEPAGE")~ and returns
the canonical clautolisp codepage spelling. Vendor
~"ANSI_NNNN"~ and clautolisp ~"WINDOWS-NNNN"~ inputs both fold
onto ~"CP-NNNN"~; Unicode names (~"UTF-8"~, ~"ISO-8859-1"~,
~"US-ASCII"~) pass through; the pre-Phase-1 empty placeholder
becomes ~"ANSI"~.

#+BEGIN_SRC autolisp
LC_ALL=fr_FR.WINDOWS-1252 clautolisp -x "(clal-system-codepage)"
"CP-1252"

LC_ALL=en_US.UTF-8 clautolisp -x "(clal-system-codepage)"
"UTF-8"
#+END_SRC

*** ~CLAL-DRAWING-CODEPAGE~ -- canonical DWGCODEPAGE
~(clal-drawing-codepage)~ is the same canonicalisation applied to
~(getvar "DWGCODEPAGE")~. At launch ~DWGCODEPAGE~ defaults to
~SYSCODEPAGE~; loading a drawing that carries its own codepage
overrides it.

*** ~CLAL-CODEPAGE-MISMATCH-P~ -- defensive comparison
~(clal-codepage-mismatch-p)~ returns ~T~ when the canonical
~SYSCODEPAGE~ and ~DWGCODEPAGE~ differ, ~nil~ otherwise. Cheap
defensive check before loading drawing-authored text: a mismatch
means strings authored under one ANSI codepage may garble when
interpreted under another.

The comparison runs on the canonicalised form, so
~"ANSI_1252"~ vs ~"WINDOWS-1252"~ correctly registers as no
mismatch.

*** ~CLAL-FILE-ENCODING~ -- BOM-sniffing file-encoding probe
~(clal-file-encoding PATH)~ opens ~PATH~, reads the first four
bytes, and returns the canonical clautolisp encoding implied by
the byte-order mark:

| BOM bytes      | Returned string |
|----------------+-----------------|
| ~EF BB BF~     | ~"UTF-8-BOM"~   |
| ~FF FE~        | ~"UTF-16-LE"~   |
| ~FE FF~        | ~"UTF-16-BE"~   |
| ~FF FE 00 00~  | ~"UTF-32-LE"~   |
| ~00 00 FE FF~  | ~"UTF-32-BE"~   |
| no BOM         | ~"ANSI"~        |

(UTF-32-LE / UTF-16-LE share the same two leading bytes;
~CLAL-FILE-ENCODING~ checks UTF-32 first so the two-byte UTF-16
prefix doesn't mask a four-byte UTF-32 BOM.)

Mirrors BricsCAD's ~VLE-FILE-ENCODING~ but preserves the
UTF-8-BOM / plain-UTF-8 distinction the BricsCAD function flattens.
A nonexistent ~PATH~ returns ~nil~ and sets ~ERRNO~ 73, matching
~LOAD~'s missing-file behaviour.

#+BEGIN_SRC autolisp
$ clautolisp -x "(clal-file-encoding \"~/.autolisp\")"
"ANSI"
#+END_SRC

*** ~CLAL-SEDIT~ -- the structural editor
~(clal-sedit [OBJECT])~ opens the *sedit* S-expression editor
(see /sedit and the aldo form navigator/) and returns the edited
object. With no argument it starts a stand-alone form at ~nil~; a
symbol recalls its recorded ~defun~/~setq~; a string is a file or
directory path; any other value is edited as a sexp. The initial and
final forms are left in ~*clal-sedit-initial-form*~ /
~*clal-sedit-last-result*~.

*** ~CLAL-CLIPBOARD-*~ -- the editor clipboard
~(clal-clipboard-copy-sexp OBJ)~ / ~(clal-clipboard-paste-sexp)~ copy
and paste a form through the in-process and system clipboards;
~(clal-clipboard-put-text S)~ / ~(clal-clipboard-get-text)~ are the
plain-text forms. Paste never evaluates its input. The system
clipboard auto-selects an X11 / Wayland / macOS / Windows / OSC 52
backend and degrades to in-process only
(=clautolisp/documentation/clipboard-interface.org=).

*** ~CLAL-NAV-*~ / ~CLAL-BREAK~ -- the debugger surface
~(clal-break)~ and ~(clal-invoke-debugger [MSG])~ enter aldo
programmatically; ~(clal-nav-function 'NAME)~,
~(clal-nav-file PATH [LINE])~ and ~(clal-nav-directory [PATH])~ open
the form navigator on a function, a file, or a directory *before* the
code runs; ~(clal-select-file PATH LINE)~ points ~list source~ at a
location; ~(clal-define-debugger-command NAMES FN [DOC])~ registers a
custom debugger command. See /Debugger/ above.

The full normative entries for these functions live in
~autolisp-spec~ §16 *clautolisp Extensions*; this section is a
quick-reference summary.
