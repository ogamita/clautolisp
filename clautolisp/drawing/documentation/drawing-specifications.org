* Scope and principles

The ~clautolisp.drawing~ package is the *substrate* under all four
drawing-facing surfaces (AutoLISP entity API, the NEW/OPEN/SAVE/CLOSE
command system, COM/ObjectDBX, and this CL API itself). Principles:

1. *Pure Common-Lisp values.* This package never traffics in AutoLISP
   wrapper types (~autolisp-ename~, ~autolisp-string~, …). A DXF datum
   is a CL integer, double-float, string, or list thereof. Wrapping
   and unwrapping is the host adapter's job. (See [REVIEW-1].)
2. *A drawing is data, not a session.* It owns entities, tables, the
   named-object dictionary, header variables, the handle allocator,
   the class table, and file provenance — nothing about prompts,
   selection sets, COM objects, or the active document. Those are host
   state.
3. *Backend-independent.* MockHost holds one or more drawings;
   LiveHost (Phase 16) may wrap a foreign database behind the same
   accessors. No function here assumes MockHost.
4. *No side effects beyond the drawing.* Mutators change the drawing
   passed in and return a useful value; they never signal document
   events (that is the host's concern, layered above).

* The value object (landed in 17a)

A ~drawing~ is a CLOS object with these accessors (all exported):

| Accessor                          | Contents                                             |
|-----------------------------------+------------------------------------------------------|
| ~drawing-name~                    | drawing name (the DWGNAME the host reports)          |
| ~drawing-path~                    | source/target absolute pathname, or NIL              |
| ~drawing-format~                  | ~:dxf-ascii~ / ~:dxf-binary~ / ~:dwg~ / ~:native~ / NIL |
| ~drawing-version~                 | format version keyword (e.g. ~:ac1027~) or NIL       |
| ~drawing-codepage~                | DWGCODEPAGE string or NIL                            |
| ~drawing-entities~                | hash: hex-handle string → ~entity-handle~            |
| ~drawing-creation-order~          | reverse-order list of handle strings                 |
| ~drawing-handle-seed~             | next-handle integer (DXF $HANDSEED)                  |
| ~drawing-tables~                  | hash: kind keyword → (hash name → ~symbol-table-record~) |
| ~drawing-named-object-dictionary~ | root ~dictionary~                                    |
| ~drawing-header-variables~        | hash: name string → ~sysvar-cell~                    |
| ~drawing-classes~                 | DXF/DWG CLASSES metadata, opaque list                |

Data carriers (also exported, re-exported by MockHost for
compatibility): ~entity-handle~, ~symbol-table-record~, ~dictionary~,
~sysvar-cell~ — unchanged from their pre-extraction shape.

* The DXF datum domain [REVIEW-1]

The single most important contract to settle. An entity's content is a
*DXF group-code list*: a list of dotted pairs ~(code . value)~ where
~code~ is an integer. Today (inherited verbatim from MockHost) the
stored list contains two *host-managed* pairs that carry AutoLISP
wrapper types:

- ~(-1 . <autolisp-ename>)~ — the entity name, an ~autolisp-ename~.
- ~(5 . "<hex>")~ — the handle string.
- ~(0 . "TYPE")~ — sometimes an ~autolisp-string~, sometimes a CL string.

That violates principle 1. The proposed canonical form for the
*drawing* layer:

- The drawing stores *only pure CL values*. Code 5 is the handle
  string; code 0 is a CL string; references (330 owner, 360, etc.) are
  handle strings. *No ~(-1 …)~ pair is stored* — the ename is a host
  notion, synthesised by the adapter from the handle on the way out.
- The host adapter (~host-entget~) wraps: it adds ~(-1 . ename)~ and
  may wrap strings as ~autolisp-string~ per dialect; ~host-entmod~ /
  ~host-entmake~ strip ~-1~ and unwrap before storing.

This moves ~normalize-entity-data~'s ename injection out of the stored
data and into the MockHost boundary. *Decision needed:* adopt the pure
form (cleaner, but the adapter does more), or let the drawing keep the
~-1~/wrapper pairs (less work now, leakier). Recommendation: pure form.

DECISION: adopt the pureform.

* API — persistence

#+begin_src lisp
(read-drawing  source &key format)            ⇒ drawing
(write-drawing drawing destination &key format version) ⇒ drawing
(probe-drawing-format source)                 ⇒ keyword | nil
#+end_src

- ~read-drawing~ — ~source~ is a pathname/namestring. Format is sniffed
  from extension then content; ~:format~ overrides. Returns a fresh
  drawing with ~drawing-path~/~-format~/~-version~/~-codepage~ set, and
  ~drawing-handle-seed~ taken from the file's HANDSEED. Signals
  ~drawing-read-error~ (a condition subclass of ~drawing-error~) on unknown format
  or parse failure.
- ~write-drawing~ — serialises ~drawing~ to ~destination~. ~:format~
  defaults to ~drawing-format~ then the destination extension;
  ~:version~ defaults to ~drawing-version~ then the codec's newest.
  Updates the drawing's provenance slots and returns it. Signals
  ~drawing-write-error~.
- ~probe-drawing-format~ — non-throwing format sniff; NIL if unknown.

Codecs register against these via an internal dispatch table; 17c adds
DXF, 17e adds DWG. The API surface does not change when a codec lands.

* API — introspection (read-only)

#+begin_src lisp
(map-entities function drawing &key include-deleted) ⇒ nil
(do-entities (var drawing &key include-deleted) &body body)          ; macro
(drawing-entity-count drawing &key include-deleted)  ⇒ integer
(find-entity drawing handle &key include-deleted)    ⇒ entity-handle | nil
(entity-dxf    entity-handle)        ⇒ list           ; the group-code list
(entity-kind   entity-handle)        ⇒ keyword        ; group-0 kind
(entity-handle-string entity-handle) ⇒ string         ; the hex handle

(drawing-table       drawing kind)            ⇒ hash-table | nil
(map-table-records   function drawing kind)   ⇒ nil
(find-table-record   drawing kind name)       ⇒ symbol-table-record | nil
(table-record-dxf    symbol-table-record)     ⇒ list

(drawing-dictionary  drawing)                 ⇒ dictionary   ; the root NOD
(dictionary-get      dictionary key)          ⇒ value | nil
(map-dictionary      function dictionary)     ⇒ nil

(drawing-variable      drawing name)          ⇒ value | nil
(drawing-variable-cell drawing name)          ⇒ sysvar-cell | nil
(map-variables         function drawing)      ⇒ nil
#+end_src

- ~map-entities~ / ~do-entities~ walk in creation order, skipping
  deleted entities unless ~:include-deleted~.
- ~find-entity~ — by hex handle string (exact match). Deleted entities
  are hidden unless ~:include-deleted~.
- Lookup folding: table names and header-variable names are
  case-insensitive; dictionary keys are case-folded strings; handles
  are matched exactly. [REVIEW-3: confirm handle case rule — DXF
  handles are uppercase hex; do we fold?]

REVIEW-3 DECISION: hexadecimal is only a representation for
integers. So unless there are other syntactic elements, handles are
integers. We can pass them as 51254306789, #xbeefea7e5, or "BEEFEA7ER",
case insensitive. For a low-level API like this I'd favor integer, but
we can add methods taking strings and parsing them as hexadecimal numbers.
Note: the use of strings in autolisp is due to their lack of bignums.

* API — mutation

#+begin_src lisp
(allocate-handle  drawing)                    ⇒ string
(add-entity       drawing dxf &key handle)    ⇒ entity-handle
(modify-entity    drawing handle dxf)         ⇒ entity-handle | nil
(delete-entity    drawing handle &key (erased t)) ⇒ boolean   ; toggle when not supplied? see note
(set-drawing-variable drawing name value &key force) ⇒ value
(add-table-record    drawing record)          ⇒ symbol-table-record
(remove-table-record drawing kind name)       ⇒ boolean
(dictionary-put      dictionary key value)    ⇒ value
(dictionary-remove   dictionary key)          ⇒ boolean
#+end_src

DECISION: I'd rather keep integers for handles. allocate-handle just
returns (prog1 handle-seed (incf handle-seed)).  Conversion to and
from strings will be done at the CL <-> autolisp interface.  Of
course, we need to define where this interface border lies. For
example, if if it goes thru DXF, and the record 5 contains a string or
an integer. It looks like it contains a string, so converting the
integer handle to hexadecimal strings is made by add-entity when it
injects ~(5 . hex-al-handle)~.  CL functions that take a handle can
accept both types thru methods or type (or integer string).

- ~allocate-handle~ — returns ~(format nil "~X" handle-seed)~ and bumps
  the seed. The sole handle source; loaders set the seed first.
- ~add-entity~ — store ~dxf~ under a fresh handle (or ~:handle~, used by
  loaders preserving file handles), append to creation order, return
  the new ~entity-handle~. Injects ~(5 . handle)~; does *not* inject
  ~(-1 …)~ (see [REVIEW-1]). Requires a ~(0 . "TYPE")~ marker; signals
  ~drawing-error~ otherwise.
- ~modify-entity~ — replace the stored data for ~handle~ (preserving the
  handle), NIL if absent/deleted.
- ~delete-entity~ — set the erased flag. [REVIEW-4: AutoLISP ~entdel~ is
  a *toggle*; that toggle is an AutoLISP-surface semantic. Should the
  CL API expose an explicit ~:erased~ boolean (proposed) and leave the
  toggle to the host adapter? Recommendation: yes — explicit here.]
- ~set-drawing-variable~ — set a header variable's value, honouring the
  cell's read-only flag unless ~:force~ (loaders pass ~:force t~ to set
  read-only header vars like $HANDSEED-derived state). Returns the
  value; NIL if the name is unknown (does *not* create cells — that is
  ~ensure-drawing-variable~, below, if we want it [REVIEW-5]).

REVIEW-4 DECISION: rename ~delete-entity~ to
~(set-entity-deleted-status drawing handle deletedp)~
and ~(entity-deleted-status drawing handle)~ ~->~ ~deletedp~
(we can also use DEFSETF to be able to write:
~(setf (entity-deleted-status drawing handle) deletedp)~
and define a TOGGLEF with DEFINE-MODIFY-MACRO
~(togglef (entity-deleted-status drawing handle))~

* Conditions

#+begin_src lisp
drawing-error            ; abstract base (subtype of error)
  drawing-read-error     ; read-drawing: unknown format / parse failure
  drawing-write-error    ; write-drawing failure
  drawing-format-error   ; unsupported/unknown format keyword
#+end_src

These are *CL* conditions, distinct from
~clautolisp.autolisp-runtime~'s ~autolisp-runtime-error~. The host
adapter translates a ~drawing-error~ into the appropriate AutoLISP
runtime error (e.g. when a ~(command "_.OPEN" …)~ fails) at the
boundary. [REVIEW-6: confirm this split.]

REVIEW-6 DECISION: split confirmed.  Also: be sure to have slots in
those conditions to find easily what drawing, what entity, etc are
concerned by the error.

* Relationship to the host / AutoLISP surfaces

- ~host-entget~ ⇒ ~find-entity~ + adapter wrapping (adds ~-1~ ename,
  dialect string wrapping).
- ~host-entmake~ ⇒ ~add-entity~ + adapter event signalling.
- ~host-entmod~ ⇒ ~modify-entity~ + events.
- ~host-entdel~ ⇒ ~delete-entity~ (the toggle lives in the adapter).
- ~host-tblsearch~ / ~-tblnext~ / ~-tblobjname~ ⇒ ~find-table-record~ /
  ~map-table-records~ (the ~tblnext~ cursor stays host state).
- ~host-getvar~ / ~-setvar~ ⇒ ~drawing-variable~ / ~set-drawing-variable~
  for *drawing-resident* sysvars; session/registry sysvars are a host
  overlay [REVIEW-7].
- ~read-drawing~ / ~write-drawing~ are what the Phase-17f NEW/OPEN/SAVE
  commands and the Phase-17g ObjectDBX ~Open~/~SaveAs~ call.

* Tooling consumers (Phase 17d), as acceptance

The API is "done" when these two tools are writable against it alone,
with no AutoLISP and no MockHost:

- ~identify-drawing FILE~ — read, walk entities + header, classify the
  drawing as SCHMS / SCHMS+ / SCHME / SCHMIEU / SCHSME+ and version.
- ~extract-schms-data FILE -o OUT.json~ — read, walk, emit JSON.

DECISION: Modulo the fact that to do that, a sizable part of the SCHMS
application code (written in autolisp) would be needed, so we'd hook
clautolisp to run it, or have to re-implement it in Common Lisp (which
would be easier probably). But some more restricted version of these
tools could do for the validation.
