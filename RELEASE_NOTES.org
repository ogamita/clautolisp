* About this file

This file records every user-visible feature shipped by the programs
that =make install= places under =$PREFIX= (default =/opt/local=).
Read it to know what each program does and where to find the full
reference. Each entry is intentionally short — one to three lines —
and points at the relevant ~user-manual.org~ for normative details.

This is not a changelog. Each section reflects the *current* state of
master at the top of the file (clautolisp 1.5.1 / alfe 1.5.1). For the
chronology and the issue-by-issue history, see ~git log~ on master and
the issues under ~issues/closed/~.

A standing rule lives in ~AGENTS.md~ § /Release notes/: this file
*must* be updated in the same commit as any user-visible feature
addition or change, including optimisations the user can observe (e.g.
faster startup, lower memory, narrower diagnostic output). Internal
refactors with no user-visible effect do not require an entry.

* clautolisp / read-autolisp (=clautolisp/documentation/clautolisp-user-manual.org=)

clautolisp is a Common-Lisp-hosted AutoLISP interpreter shipped as a
standalone CLI. read-autolisp is its tokenizer/reader available as a
separate binary for tool integrators.

** Standalone CLI (clautolisp-sbcl / clautolisp-ccl)

- Three modes: file (~clautolisp file.lsp~), expression (~-x EXPR~),
  and interactive REPL. See § /Modes of Operation/.
- REPL comma-commands: a line starting with =,= is a command, not an
  expression — =,d=/=,date=, =,u=/=,uptime=, =,h=/=,help=, =,q=/=,quit=
  (exit even when ~(quit)~ is not available). An empty line just
  re-prompts. Backed by the interactor framework's command dictionaries
  (the debugger dictionary mechanism, generalised). See § /Modes of
  Operation / Interactive REPL/.
- Session dribble: ~--dribble~ / ~--dribble=FILE~ (or ~(clal-dribble
  [file [interactors]])~ at runtime, a toggle) records the REPL
  interactions into an append-mode log — input raw, output/error
  lines as =;; O:= / =;; E:=, condition reports as =;; C:=, under a
  =;; H: clautolisp VERSION= header. Filtered by interactor:
  ~AUTOLISP~ only by default (a debugger/sedit stop pauses the
  recording); ~--dribble-interactors=t|NAMES~ /
  ~*CLAL-DRIBBLE-INTERACTORS*~ widen the set. Default file:
  =$XDG_STATE_HOME/clautolisp/dribbles/YYYYMMDDTHHMMSS.log=. See
  § /Modes of Operation / Interactive REPL / Recording the session/.
- Five dialects selectable via ~--strict~ / ~--autocad~ /
  ~--bricscad~ / ~--clautolisp~ / ~--lax~. The ~lax~ dialect silences
  every encoding-dispatch diagnostic globally. See § /Options /
  Dialect selection/.
- ~(setq *AUTOLISP-DIALECT* 'DIALECT)~ switches the dialect live: the
  REPL reader and the debugger's / inspector's form reads consult the
  variable at each read (one shared reader entry,
  ~issues/open/interactor-design-revision.issue~ D2), so a mid-session
  switch takes effect on the very next line.
- Host backend selectable via ~--host mock|null~ (default mock).
- ~vl-registry-read/-write/-delete/-descendents~ are host-backed and
  *persistent* on the mock host ($XDG_CONFIG_HOME/clautolisp/
  registry.sexp — the Windows-registry / macOS-defaults emulation):
  values survive across runs, keys are case-insensitive backslash
  paths, a nil value-name is the key's default value, and
  ~vl-registry-descendents~ returns immediate sub-keys by default
  (value names with the flag).
  Mock-host carries the full 1836-entry vendor sysvar catalogue.
- Under ~--bricscad~ the sysvar *set* matches BricsCAD: the 382
  catalogue variables BricsCAD does not define (confirmed by both the
  inventory and a live BricsCAD V26 probe) are dropped, so ~(getvar
  "DYNPROMPT")~ returns ~nil~ and ~(setvar …)~ errors, as on a real
  BricsCAD. Other dialects keep the full AutoCAD-derived catalogue.
  Engine identity (~PROGRAM~ / ~ACADVER~ / …) and the dynamic codepage
  sysvars are unaffected. See
  ~issues/open/bricscad-dialect-sysvar-parity.issue~.

** Encoding handling (issues/closed/encoding-dispatch.issue)

- Launch chain: =LC_ALL → LANG → LC_CTYPE → -e= resolves the active
  encoding at startup and populates ~SYSCODEPAGE~, ~DWGCODEPAGE~, and
  ~*AUTOLISP-FILE-ENCODING*~ (no more empty placeholders). Empty-
  string sentinel for ~*AUTOLISP-FILE-ENCODING*~ in the degenerate
  case (no locale env, no -e) falls through to the dialect default.
  See § /Source-file and terminal encoding/.
- ~LOAD~ and ~OPEN~ accept a positional ~[encoding]~ argument under
  ~--clautolisp~ / ~--lax~. Foreign-dialect spellings (AutoCAD's
  ~"utf8"~ / ~"utf8-bom"~ third arg, BricsCAD's ~,ccs=…~ mode suffix,
  ~LISPSYS~) work under their native dialect; other dialects emit
  ~enc-foreign-dialect~ but still honour the call.
- Diagnostic codes: ~enc-foreign-dialect~, ~enc-extension-used~,
  ~enc-unsupported-target~, ~enc-invalid-value~,
  ~enc-lispsys-out-of-range~, ~enc-codepage-mismatch~,
  ~enc-unknown-codepage~, ~enc-host-dependent~. See § /Errors and
  Diagnostics / Encoding-dispatch diagnostics (enc-*)/.

** clautolisp-only helpers (CLAL-* prefix)

- ~clal-command-log~ — the ~command~ / ~command-s~ / ~vl-cmdf~ token
  sequences routed to the host this session, oldest first. Under the
  MockHost (no command engine) the recorded log is the whole
  observable effect of a ~command~ call — the deterministic
  assertion surface for headless rigs. See § /clautolisp
  Extensions/.
- ~clal-sysvar-list~, ~clal-sysvar-apropos~ — discover the active
  host's system variables.
- ~clal-system-codepage~, ~clal-drawing-codepage~,
  ~clal-codepage-mismatch-p~ — canonicalised SYSCODEPAGE / DWGCODEPAGE
  query for defensive comparison.
- ~clal-file-encoding PATH~ — sniff the BOM, return one of
  ~"UTF-8-BOM"~ / ~"UTF-16-LE"~ / ~"UTF-16-BE"~ / ~"UTF-32-LE"~ /
  ~"UTF-32-BE"~ / ~"ANSI"~. See § /clautolisp Extensions/.

** sedit — structural editor + clipboard (=clautolisp/documentation/clautolisp-sedit-spec.org=)

- ~(clal-sedit [object])~ — edit a form with the *sedit* structural editor: no
  argument starts at ~nil~; a symbol recalls its recorded definition; a string
  is a file or directory path; any other value is edited as a sexp. Two-mode
  (navigate / edit) with motions, insert/add/replace, wrap/slurp/barf/splice/
  split/join, one-level undo, and byte-preserving formatting. Returns the edited
  object. See the new manual chapter § /sedit and the aldo form navigator/.
- The interactive REPL now records every top-level form it evaluates, so
  ~(clal-sedit 'NAME)~ recalls a function/variable you defined at the prompt.
- *Clipboard* (system-clipboard aware, degrades to in-process only):
  ~(clal-clipboard-copy-sexp OBJ)~ / ~(clal-clipboard-paste-sexp)~ and the text
  forms ~(clal-clipboard-put-text S)~ / ~(clal-clipboard-get-text)~. Paste never
  evaluates its input. Providers: X11, Wayland, macOS, Windows, OSC 52, NULL.
- From the aldo form navigator, an editing word (=edit=, =insert=, =wrap=, …)
  opens the same editor on the current function; leaving it installs the edited
  definition in the running system (spec §7).
- A bare Lisp *form* typed at any prompt — =DBG>=, the form navigator =NAV>=, or
  the editor =SEDIT>= — is evaluated and its result printed, like the REPL.
- =,settings NAME= now reports an error for an unknown setting name instead of
  printing a bogus value.
- The public =*clal-*= extension variables (=*clal-clipboard*=,
  =*clal-sedit-initial-form*= / =*clal-sedit-last-result*=, the =*clal-form*= /
  =*clal-result*= / =*clal-source-form*= eval-history trio and their =-1= / =-2=
  generations, =*clal-on-error*=, =*clal-aldo-configuration*=) are now *bound* at
  startup, so AutoLISP code can read them and =(alref-apropos "*CLAL-")= lists
  them; =clal-clipboard-copy-sexp= and =clal-clipboard-put-text= both set
  =*clal-clipboard*= (so it reflects the write even when the system clipboard is
  unavailable or does not persist the value), and =clal-sedit= sets
  the sedit variables. Each sedit evaluation (=e=, =macroexpand=, or a bare form
  at the prompt) *shifts* the history variables (spec §5.6): =*clal-form*= /
  =*clal-source-form*= / =*clal-result*= hold the most recent, with the =-1= /
  =-2= generations holding the two before — a clautolisp analogue of Common
  Lisp's =+= / =*= REPL variables.

** aldo debugger — form navigator (=clautolisp/documentation/clautolisp-sedit-spec.org=)

- A navigated form with *no source file* (e.g. one defined with
  ~clal-compile~) is now laid out by the AutoLISP indentation rules — newlines
  by special operator, two-space body indent (sedit-spec §5.2, the same layout
  sedit gives it) — with the selection marked, instead of one flat line. The
  same applies to sedit's own selection display: the spine down to a deep
  selection keeps the layout instead of flattening (=format-marked=, =reflow=
  in the =clautolisp.sedit= API).
- The navigator no longer *re-renders the whole view after every command*: an
  output-only turn (a stacked debugger command, an evaluated form, help) just
  prints its output; a bare =RET= redisplays. Motions and edits still redraw.
- With =set navigator sexp= (the default) *every stop* — breakpoint, step,
  error — now enters the sexp form navigator directly, anchored on the current
  poll-point (sedit-spec §7): no =nav= command needed. The flat source-line
  window (the *line* navigator's display) no longer also prints there; =q=
  drops to the flat =DBG>= command loop. With =set navigator line= the stop
  keeps the flat window at =DBG>=.
- Entering the form navigator (=nav=, =g NAME=, =(clal-nav-function 'NAME)=)
  now selects the function's *first poll-point*, not the whole ~defun~ head —
  and, when entered on a *break*, re-anchors to the *current* poll-point (the
  form the debugger stopped on), not the first (sedit-spec §7 re-entry).
- The form navigator's prompt is now plain =NAV>= in every case, never
  =NAV[code]>= / =NAV[non-code]>= — one prompt per mode (=DBG>= debugger, =NAV>=
  navigator, =SEDIT>= editor; sedit-spec §6). A bare form evaluated at the =NAV>=
  prompt is echoed with =NAV>= too, not =DBG>=.
- In navigation mode the navigator keys (=d u > < << >> ±N=) are stacked over
  the full debugger dictionary: every non-motion command — =c i n o a r j=,
  the breakpoint / frame / inspection verbs — dispatches directly, and =?=
  lists them. (The former =i k jj j l ll= letter aliases are retired: they
  shadowed debugger verbs.)
- Interactor routing in the navigator: =aldo CMD= / =debug CMD= (=debug= is
  the debugger interactor's alias) route CMD to the debugger explicitly —
  dispatching normally, user commands first (e.g. =aldo b LOC= while bare
  =b= breakpoints the selection) — the same prefix the sedit editor uses.
- The debugger's built-in vocabulary is now itself a command dictionary
  (the interactor framework): user commands live in a separate dictionary
  layered over it, so they can shadow any built-in.
- One AutoLISP registration API for every interaction mode
  (~issues/open/interactor-design-revision.issue~ D6/D7):
  =(clal-define-command "INTERACTOR" NAMES FN [DOC])= registers a user
  command of the named interactor's user dictionary —
  =(clal-list-interactor-names)= lists the valid names (=ALDO=, =NAVI=,
  =LAVI=, =INSPECT=, =AUTOLISP=) — and commands install before their
  interactor is activated. There is no global user table: a "global"
  command is one defined on the =AUTOLISP= interactor at the bottom of
  the stack. =clal-define-debugger-command= is deprecated, the
  equivalent of =(clal-define-command "ALDO" …)=.
- The REPL is the =AUTOLISP= interactor (alias =LISP=), and it really
  is at the bottom of the stack: at a debugger stop the whole ambient
  stack stays reachable, so a user command registered on =AUTOLISP=
  dispatches from =DBG>= / =NAV>= by fall-through, and =lisp CMD= /
  =autolisp CMD= route to it from any inner mode (at the REPL itself,
  invoke it as =,CMD=).
- sedit is the =SEDIT= interactor (sedit-spec §6.6 amendment, Option A):
  ONE dictionary — motions over *all* sub-sexps plus the editing
  commands — under a single =SEDIT>= prompt; the internal NAV/EDIT
  two-mode machine is gone. =(clal-sedit …)= pushes it over the REPL
  and returns the edited object when =q= pops it; an editing word at
  =NAV>= pushes it over the navigator and the stop. SEDIT's keys shadow
  same-key debugger verbs (=aldo CMD= reaches them); unshadowed
  commands fall through; =aldo c= from inside the editor resumes.
- Quitting *at a stop* now asks: =q= at =NAV>= or =SEDIT>= above the
  debugger warns that quitting aborts the debugged execution, lists the
  resume alternatives, and asks =really abort? (y/n)= — =y= runs the
  debugger's own =quit=, anything else keeps debugging
  (interactor-design-revision.issue T4).
- The =command= word (like bash's =command foo=) is the systematic escape to
  a shadowed built-in: =command CMD= / =aldo command CMD= skip the user
  commands — how a shadowing user command calls the system one. =debugger
  CMD= remains as the historical spelling. See the command reference §8
  /Command dictionaries (stacked dispatch)/.
- A wrong command no longer drops the debugger
  (=issues/closed/error-while-debugging.issue=): at an *error stop*,
  =i=/=n=/=o=/=a=/=j= refuse with the §10.1 guidance instead of silently
  declining the stop (which let the error unwind to the toplevel handler);
  an error escaping any command body is reported (=error while executing
  CMD=) and the stop keeps reading — a debugging state is only dropped by
  an explicit =q=. An AutoLISP error raised while a command evaluates user
  code still breaks into a nested debugger stop first.
- =r='s expression is now *mandatory* (every AutoLISP function returns
  something): a bare =r= prompts =return value?=; a blank answer cancels.
- Correctness after a *caught* error
  (=issues/closed/usubr-arity-error-pops-callers-frame.issue=): a caught
  wrong-number-of-arguments error (=vl-catch-all-apply=, a debugger
  resume) used to pop the *caller's* dynamic frame, so the caller's
  variables silently resolved to stale outer values from then on (the
  reported infinite =fact= recursion). The frame push/pop now pairs
  exactly.
- One loop for every interaction mode (=issues/open/interactor-unification.issue=):
  the debugger prompt, the form navigator, the line navigator, the
  inspector and the REPL all run on the interactor framework's single
  INTERACTOR-LOOP, with their vocabularies as stacked command
  dictionaries. User-visible effects: from the *line* navigator and the
  *inspector* every unshadowed debugger command now works directly (the
  spec §8 stacked dispatch — e.g. =c= continues from inside the
  inspector; its =b= stays =bind=), a bare =(FORM)= evaluates in the
  inspector too, ±N is the =skip= command everywhere, and =aldo CMD= /
  =command CMD= routing works in every mode. Inspector arguments are
  whitespace-separated (=d 0=, =e (+ 1 2)= — no glued =d0=).
- =(clal-nav-function 'NAME)= at the REPL opens the navigator *without* faking
  a break, so you can browse a function and place breakpoints before it runs.
- *Virtual breakpoints* on a file *not loaded yet*: browsing such a file
  (=(clal-nav-file PATH)=, or descending from =(clal-nav-directory)=) and
  pressing =b= on a form records a virtual breakpoint — kept *relative to its
  top-level =defun=* so it survives the file being edited above the form —
  that is *armed automatically* when the file is loaded and the function
  instrumented, firing on the very first post-load run. =lb= lists pending
  records as =vbN=; =delete vbN= removes one. See the command reference §2
  /Virtual breakpoints/.
- The directory browser's =directory-listing= setting now honours =column=:
  an ls -C-style column-major grid (=short= = ls -1 and =long= = ls -l as
  before), still under =directory-window= elision. See the command
  reference §8.
- Setting a breakpoint twice on one poll-point is idempotent — one breakpoint,
  one decoration ("already set" on the repeat). See the three
  =issues/open/bug-aldo-*= tickets.
- *Debugger CLI options* (=issues/open/debugger-public-interface-and-on-error.issue=
  Parts B-D; user manual § /Options / Debugger (aldo)/): the three event
  policies =--on-error quit|debug|ignore= (default: =debug= at the
  interactive REPL, =quit= in batch), =--on-interrupt debug|ignore|quit=
  (default =debug=: Control-C breaks into aldo at the interrupt point;
  =ignore= resumes; =quit= exits 130; a second C-c while one is handled
  exits immediately), and =--on-quit debug|quit= (default =quit=; =debug=
  enters aldo from ~(quit)~ / ~(exit)~ *before* the stack unwinds —
  continue quits, abort cancels). Each is mirrored by an AutoLISP
  variable — =*clal-on-error*=, =*clal-on-interrupt*=, =*clal-on-quit*= —
  and the interrupt / quit policies are re-read *live*, so ~(setq …)~
  overrides the CLI at runtime.
- *Debugger UI selection*: =--debugger-ui tui|ncurses|aldb= is the per-run
  override of the persisted =default-user-interface= aldo setting
  (=aldo.conf= is now consulted at launch); =--aldb-listen [HOST:]PORT= /
  =--aldb-stdio= configure the aldb (Emacs) channel and imply
  =--debugger-ui aldb= (the aldb RPC transport itself is still pending —
  selecting it fails with a clear message). These REPLACE the former
  =--aldo-user-interface= / =--aldb-listening-address= /
  =--aldb-listening-port= spellings (no aliases kept).

** Exit-status channel (portable pass/fail for batch scripts)

- ~(autolisp-set-status N)~ records an integer process exit status;
  ~(autolisp-status)~ reads it back (0 when never set). The name and
  integer semantics match the ~autolisp-set-status~ alfe injects into a
  driven AutoCAD / BricsCAD session, so one ~C:MAIN~ reports pass/fail
  identically across all three engines.
- ~(quit [status])~ / ~(exit [status])~ take an OPTIONAL integer: ~(quit
  N)~ exits ~N~, bare ~(quit)~ exits with the recorded status. The
  clautolisp CLI's normal end-of-run also exits with the recorded status,
  so ~-x '(C:MAIN)'~ need not call ~quit~ at all.
- These are clautolisp extensions: silent under ~--clautolisp~ / ~--lax~,
  and flagged with a non-fatal out-of-dialect WARNING under ~--strict~ /
  ~--autocad~ / ~--bricscad~ only when the status argument is supplied.
  See ~issues/closed/autolisp-set-status-and-quit-status.issue~.

** Surface coverage of the AutoLISP / Visual LISP spec

- 100% surface-bound: 548 real implementations plus 410 M6 stubs
  (nil-returning placeholders) under every documented identifier.
  Calling a stub returns nil; ~(boundp 'X)~ is ~T~. Real impls track
  through ~issues/open/deferred-stubbed-functions.issue~. ~ALERT~
  is now a real builtin (emits ~"WARNING: <message>"~ to stdout in
  the headless engine).

** Other AutoLISP semantics matches

- ~command~ / ~command-s~ are real special forms routed through the
  Host Abstraction Layer: arguments evaluate, values normalize to
  command-line token strings (points as ~"X,Y[,Z]"~, ~""~ = RETURN,
  ~"\\"~ = PAUSE — the predefined ~pause~ variable is bound), and
  the sequence lands on the host command channel; MockHost records
  it and returns nil. ~vl-cmdf~ shares the routing and returns T on
  success. See the user manual § /Host backend/.
- ~(quit)~ and ~(exit)~ are catchable by ~VL-CATCH-ALL-APPLY~ — the
  vendor surface, needed by test frameworks like test-unitaire.lsp.
- Source-level ~'NIL~ normalises to CL nil so ~(a b c . NIL)~ is a
  proper list (REVERSE / LENGTH / MAPCAR etc. accept it).
- Reader accepts ~"42."~, ~"5."~, ~".5"~ as reals (vendor semantics,
  distinct from CL).
- Float printer emits ~"3.14"~ / ~"42.0"~ / ~"1.0e20"~ — no CL
  double-float marker.
- Relational operators ~<~, ~<=~, ~>~, ~>=~ fold non-numeric args
  (including nil) to nil — matches AutoLISP loop-guard idioms.
- ~(substr s start 0)~ returns ~""~.
- Locale-insensitive numeric I/O (period decimal separator regardless
  of ~LC_NUMERIC~ / ~DIMDSEP~).

* alfe — AutoLisp FrontEnd (=autolisp-front-end/documentation/alfe-user-manual.org=)

alfe is the multi-backend driver: it speaks the AutoLISP CLI surface,
resolves a backend (in-process clautolisp, or a CAD-host bridge via
file or COM), and forwards the workload.

** Backends (autolisp-front-end/documentation/alfe-user-manual.org § /Backends/)

- ~--clautolisp~ (default) — drive the in-process clautolisp engine.
- ~--bricscad~ — drive a BricsCAD process via the file-protocol
  bridge (BricsCAD V26 / macOS). Output a script directs to a file it
  opened — ~(princ s f)~, ~(print s f)~, ~(prin1 s f)~ on a descriptor
  from ~(open path "w")~ — now lands in that file instead of leaking
  onto the captured stdout channel.
- ~--autocad~ — drive an AutoCAD process (Windows + macOS lanes).
- ~--echo~ — a test backend that logs without executing.
- On MS-Windows, ~--bricscad~ and ~--autocad~ now discover installed
  BricsCAD / AutoCAD binaries from the standard ~Program Files~ roots
  by default; ~BRICSCAD_EXE~, ~AUTOCAD_EXE~ and
  ~AUTOCAD_ACCORECONSOLE~ still override discovery explicitly.
- ~--mode~ and ~--dwg~ are now forwarded correctly into CAD backend
  startup, so ~alfe --autocad --mode batch --dwg file.dwg ...~ no
  longer falls back silently to ~:auto~ / automation mode.
- ~alfe --autocad --mode batch~ now falls back to
  ~AUTOLISP_DWG~ or a shipped ~acadiso.dwt~ / ~acad.dwt~ template
  next to the discovered AutoCAD install, instead of requiring an
  explicit ~--dwg~ on every invocation.
- AutoCAD batch bootstrap now reports the child ~accoreconsole~
  process exit code and console diagnostics immediately when the CAD
  process dies before publishing ~READY~, instead of timing out with a
  generic ~READY-TIMEOUT~.

** Modes

- ~--mode auto~ (default), ~--mode batch~, ~--mode automation~.
  See § /Backend invocation/ for the file-protocol handshake.

** Diagnostics under ~--verbose~ / ~--debug~

- Both flags now emit progressive diagnostics through every
  backend's lifecycle (workdir, argv, BOOTSTRAP timing, status-file
  polling). The old "single error line" failure mode is gone.

** Image-introspection variables

- alfe publishes the same ~*AUTOLISP-…*~ globals as clautolisp, plus
  ~*AUTOLISP-MODE*~, ~*AUTOLISP-DRAWING*~, ~*AUTOLISP-WORKDIR*~,
  ~*AUTOLISP-KEEP-WORKDIR*~. See the alfe manual's
  § /The *AUTOLISP-…* globals/ table.

* autolisp-benchmark — performance suite (=autolisp-benchmark/documentation/autolisp-benchmark-user-manual.org=)

- A pure-AutoLISP benchmark suite that times seven operation classes —
  arithmetic, lists, strings, serialization (object <-> string),
  file-io, entity (CAD create/modify/delete), and memory-gc — each for
  about five seconds, and reports throughput (iterations/second) and
  cost (microseconds/iteration) per class plus an overall total.
- Loaded into any engine on the same hardware for a direct comparison:
  ~clautolisp -l autolisp-benchmark/harness/run.lsp -x
  '(autolisp-benchmark-run-all)'~, or via ~alfe --bricscad~ /
  ~--autocad~ / ~--clautolisp~. Convenience targets: ~make -C
  autolisp-benchmark run~ (clautolisp), ~run-bricscad-*~,
  ~run-autocad-*~; per-class budget via ~BENCH_SECONDS=~.
- Auto-detects the implementation via ~(ver)~ and the clock via
  ~(getvar "MILLISECS")~ (with a ~DATE~ fallback), relying on the live
  timer sysvars of the clautolisp 1.3.x line, so results are labelled
  and comparable across targets. Installed under
  ~$PREFIX/libexec/autolisp-benchmark/~ (harness + benchmarks + docs).
