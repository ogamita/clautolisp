Decisions (see clautolisp + alfe specs for full prose):

* LC_ALL > LC_CTYPE > LANG is consulted for the source-file
  default encoding ONLY (the bytes-on-disk -> characters
  interpretation). The remaining LC_* categories (LC_COLLATE,
  LC_MESSAGES, LC_NUMERIC, LC_TIME, ...) are NOT consulted for
  encoding; if/when the runtime localises messages or decimal
  separators, those features will consult their own matching
  category, per POSIX.

* Full precedence at session start, strongest first:
  1. -e ENC on the alfe/clautolisp command line.
  2. LC_ALL / LC_CTYPE / LANG encoding suffix
     (en_US.UTF-8 -> :utf-8, fr_FR.ISO-8859-1@euro -> :iso-8859-1).
  3. Dialect default (strict -> :iso-8859-1; autocad-2026 /
     bricscad-v26 -> :utf-8).

* Terminal I/O encoding is owned by the underlying Lisp (SBCL/CCL)

  was needed there; AutoLISP (princ ...) routes through the host

# The LC_* Environment Variables

The `LC_*` variables control **locale** behavior on Unix-like systems — how
programs format and interpret things like dates, numbers, currency, text
sorting, and character encoding. Each variable governs a specific category,
and together they let you mix and match conventions (e.g., English messages
with French number formatting).

## The categories

The main `LC_*` variables and what each controls:

- `LC_CTYPE`         — Character classification and encoding. Determines what
                       counts as a letter, digit, uppercase/lowercase, and
                       which character set is in use (e.g., UTF-8). This is
                       often the most consequential one in practice.
- `LC_COLLATE`       — String comparison and sorting order. Affects `sort`,
                       `ls`, and any program comparing strings. (For example,
                       whether `ä` sorts near `a` or after `z`.)
- `LC_MESSAGES`      — Language of program messages, prompts, and yes/no
                       responses.
- `LC_NUMERIC`       — Formatting of non-monetary numbers: decimal point
                       (`.` vs `,`) and thousands separator.
- `LC_MONETARY`      — Currency symbol, monetary decimal/grouping separators,
                       positive/negative sign placement.
- `LC_TIME`          — Date and time formatting: month/day names, 12 vs
                       24-hour clock, date order.
- `LC_PAPER`         — Default paper size (Letter vs A4).
- `LC_NAME`          — Format for personal names (salutation, given/family
                       order).
- `LC_ADDRESS`       — Format for postal addresses and country representation.
- `LC_TELEPHONE`     — Format for telephone numbers.
- `LC_MEASUREMENT`   — Units (metric vs imperial).
- `LC_IDENTIFICATION`— Metadata about the locale definition itself (rarely
                       user-facing).

## Priority order

When a program needs to know, say, how to format a date, it resolves the
locale in this order:

1. `LC_ALL` — If set, this overrides everything. It forces every category to
   the same locale, ignoring all individual `LC_*` variables and `LANG`.
   Useful for scripts that need predictable behavior (`LC_ALL=C` is the
   classic example).
2. The specific `LC_*` variable for that category (e.g., `LC_TIME` for date
   formatting).
3. `LANG` — The fallback default for any category not otherwise set. Think of
   it as "the locale for everything I haven't specified."
4. The `C` (or `POSIX`) locale — A built-in minimal default if nothing above
   is set: ASCII, English, byte-wise sorting, no fancy formatting.

So the resolution for any one category is:
    LC_ALL  →  LC_<category>  →  LANG  →  C

## What happens if one is missing

"Missing" just means unset, and the fallback chain handles it transparently
— there's no error. If `LC_TIME` is unset, dates fall through to `LANG`; if
`LANG` is also unset, they fall through to `C`. Each category resolves
independently, which is why you can run with `LANG=en_US.UTF-8` and set just
`LC_TIME=de_DE.UTF-8` to get English everything but German-formatted dates.

One important gotcha: `LC_ALL` is sticky. If it's set (even to an empty
string in some shells, though usually only a non-empty value counts), it
wins, and setting `LC_TIME` will have no visible effect until you unset
`LC_ALL`. This catches people out constantly — you export `LC_TIME` to fix
date formatting, nothing changes, and the culprit is a lingering `LC_ALL`
from somewhere upstream.

You can inspect the current state with the `locale` command, which prints
every category and the value it resolved to, and `locale -a` lists the
locales actually installed on the system. If you ask for a locale that
isn't installed (say `fr_FR.UTF-8` on a stripped-down container), programs
typically warn and fall back to `C`.
